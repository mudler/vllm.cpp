# Backend and architecture matrix

This is the canonical platform, CUDA-target, and native-competitor inventory for
roadmap D1. The accepted spike is
[specs/cuda-architecture-inventory.md](specs/cuda-architecture-inventory.md).
Upstream references are pinned to vLLM `e24d1b24`; local references describe
`vllm.cpp` at the inventory baseline `f7ccaa7`.

Build availability is not runtime support. A target remains `INVENTORIED` until
it has a native build, ported tests, hardware execution, and a trace showing the
intended dispatch. Performance claims additionally require same-workload runs
under [benchmark-protocol.md](benchmark-protocol.md).

**How an architecture is added (the additive contract, `BACKEND-CUDA-ARCH-ADDITIVITY`).**
Since [arch-additivity](specs/cuda-arch-additivity.md) landed, arch support is
declared as DATA in two places and nowhere else: widen the relevant cells of
`VT_CUDA_FEATURE_TABLE` in [cmake/CudaArchFeatures.cmake](../cmake/CudaArchFeatures.cmake)
(each feature is resolved per arch against the requested target list, so a fat
build can no longer silently drop a capability), and register the arch's kernel
tactic from its own translation unit via
[cuda_arch_tactics.h](../src/vt/cuda/cuda_arch_tactics.h) (host launchers select
at runtime from the cached device capability and are never edited). This makes
the expansion mechanical; it does NOT confer support. Exactly ONE tactic is
registered today — the existing `sm_12x` fp4 path. Two rows have moved off
`INVENTORIED` to a **BUILD-supported** (not runtime) state: `BACKEND-CUDA-SM120`
(same-family consumer Blackwell — shares the sm_12x kernel bodies, so it carries
ALL fast paths) and `BACKEND-CUDA-SM090` (Hopper — the first CROSS-FAMILY arch,
single-arch, but PORTABLE-KERNELS-ONLY: it has NO Hopper fast-path body, so every
fp4/CUTLASS/Marlin/FA feature resolves EMPTY). Neither has run a gate model — no
sm_120 or sm_90 board exists here. A vLLM-competitive Hopper path, and any
cross-family FAT build, still need the per-arch kernel bodies plus per-source
gencode narrowing: a kernel campaign, not an additive drop-in.

|---|---|---:|
| NVCC `< 12.8` | `7.0, 7.5, 8.0, 8.6, 8.7, 8.9, 9.0` | 7 |
| NVCC `>= 12.8, < 13.0` | `7.5, 8.0, 8.6, 8.7, 8.9, 9.0, 10.0, 10.1, 10.3, 12.0, 12.1` | 11 |
| NVCC `>= 13.0` | `7.5, 8.0, 8.6, 8.7, 8.9, 9.0, 10.0, 11.0, 12.0` | 9 |

The union is exactly 13 numeric targets. vLLM extracts PyTorch gencodes and
removes inherited `+PTX` (`CMakeLists.txt:201-220`), so a component must request
PTX explicitly. `cmake/utils.cmake:294-345` emits both SASS and PTX for
`+PTX`; `:348-485` implements loose target intersection. An `f` target matches
the same major family, an `a` target preserves architecture-specific features,
SASS fallback stays inside the major family, and PTX may cross major families.
The SM90+ helper is `cmake/utils.cmake:488-495`.
