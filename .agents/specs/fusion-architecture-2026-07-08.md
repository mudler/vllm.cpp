# RECOMMENDED FUSION ARCHITECTURE — "Tiered Declarative Recipe" (TDR)

## 0. The framing the five judges converged on

The 20 confirmed prefill gaps split into **three structurally distinct classes**, and no single mechanism spans them. Any honest architecture must fence itself to one class:

- **Class A — standalone HBM-pass chains** (fused_add_rms_norm, rmsnorm+fp4-quant, silu·mul+fp4-quant, qk-norm+rope+gate, MoE combine+gate, GDN post-conv glue). Row-structured, bandwidth-bound. **This is the only class a portable fusion framework can own.**
- **Class B — GEMM-epilogue chains** (dequant-scale+bias+act+cast+residual). Must fold into the GEMM; CUDA = cutlass EVT, which is compile-time template metaprogramming with **no Vulkan/Metal analog**. Grounded check: our GEMMs today (`cuda_matmul_fp8_cutlass.cu`, `cuda_matmul_nvfp4_cutlass.cu`) use only the auto-builder `LinearCombination`/`alpha_ptr` epilogue — **no EVT visitor tree exists in-tree**. → **out of the recipe framework, its own cutlass-EVT track behind existing GEMM ops/levers.**
- **Class C — kernel-internal** (FA-2 cp.async/softmax-log2, GDN `v_new` memset + DeltaH/ChunkO tiling, fp4-GEMM tactic/StreamK scheduler). **Not fusions at all** → their own tracks (Tasks 16/18/21/23).

Grounding the payoff denominator (critical, and it reshapes the recommendation): of the 20 gaps, only ~8–10 are Class A, **and most are already shipped as hand ops** — `AttnQkNormRopeGate` (`cuda_ops.cu:390`), `RopeCosSinCache`, `SiluMulFp4Quant` (`ops.h:327`), `RmsNormResFp4Quant` (Task 11 done), `MoeCombineGate`, GDN glue (`GdnPostConv/GdnGBeta/GdnConvSplit`). Several are **default-OFF pending GPU gating** (`VT_FUSE_SILU_QUANT`, `VT_FUSE_ATTN_PREAMBLE`). So on CUDA **today**, a fusion framework is mostly *refactor + safe-flip-on*, not net-new perf. Its real value is **structural** (retire the growing bespoke-kernel surface, kill CPU/CUDA parity drift) and **strategic** (backend #2 inherits fusions cheaply) — the two axes the user weighted highest.

## 1. The architecture

TDR combines candidate #1's POD recipe (the declaration), #3's declared-vs-realized seam (correctness), and #2's single-source-of-truth + build-time specialization (perf) — with a **graceful tier degradation** that fixes each candidate's fatal flaw.

**Declaration — one recipe, above vt:: (the WHAT).** A fusion is a `constexpr FusedRecipe` POD (candidate #1's opcode list + operand bindings) in a new `include/vt/recipes.h`, built on a new `include/vt/fused_recipe.h`. Each builder transcribes a vLLM fusion *pattern* — `vllm/compilation/passes/fusion/{rms_quant_fusion,act_quant_fusion,qk_norm_rope_fusion}.py` — as an opcode list. We lift the WHAT, never a CUDA kernel. The recipe lives ABOVE vt:: (model forward includes `recipes.h`), so it is the **single source of truth** shared by every realization tier — this is what kills the CPU/CUDA drift the codebase carries today (the `ops.h:327` "CPU fallback = composite sequence" is a hand-kept parallel of the CUDA kernel).

**Realization — per backend, three tiers that degrade gracefully (the HOW):**

- **Tier 0 — composite over existing vt:: ops (correctness, every backend, FREE).** The recipe's step list is walked op-by-op through the *already-registered* primitives (`RmsNorm`, `SiluAndMul`, `RopeNeox`, …). This is candidate #3's declared-vs-realized seam and it makes a new backend **correct on day 1** and hands the CPU golden-oracle every present + future fusion for free. Crucially this is the **default**, not an error path — fixing candidate #4's fatal flaw (`GetOp` throws on an unregistered op → hard crash; `ops.cpp:29`).
- **Tier 1 — ONE interpreter kernel per backend (portable speed).** Candidate #1's single kernel that walks the recipe in one HBM pass over a small per-thread virtual register file. This is where **Vulkan/Metal get every Class-A fusion cheaply** — one kernel port per backend, not one kernel per fusion. Captures the full HBM-round-trip elimination, which *is* the entire win for these bandwidth-bound chains.
- **Tier 2 — build-time structural specialization (hand-fused parity for the measured-hot few).** Candidate #2's compiler-instantiation of a shared archetype skeleton: the recipe drives a *templated* kernel so the register file is static (no CUDA local-mem spill — candidate #1's fatal flaw). On CUDA this **reuses the existing `AttnQkNormRopeGateKernel` block-reduce skeleton** (`cuda_ops.cu:390-455`) as the archetype. Reserved for recipes where Tier-1 measurably loses to hand-fused.

**The quant terminal is backend-negotiated, not portable (accepted, per `backends.md`).** `QUANT_BLK_FP4/FP8` are NVIDIA-specific; Metal/Vulkan/XPU use GGUF k-quants. So the **norm/act/rope/gate prefix ports** across backends via Tiers 0/1; the **quant tail is a per-backend opcode** (+1 switch case). This is candidate #1's fatal flaw handled honestly rather than hidden.

**Where it sits on the vt:: seam.** Declaration above vt::. Realization AT the existing seam: one new `OpId::kFusedChain` dispatched through the **exact** `OpId`/`Fn`/`RegisterOp`/`GetOp` table every op already uses (`ops.cpp:20-33`); Tier-2 specializations register as additional table entries keyed by recipe id. Zero new dispatch machinery.

**How a fusion is declared once + realized everywhere:** declare = 1 `constexpr` builder (~10 lines) in `recipes.h` + 1 call-site swap in `qwen3_5.cpp`. Realize = **nothing per backend** for Tier 0 (composite) and Tier 1 (the one interpreter already handles any recipe). Only a genuinely-new *primitive* opcode (new activation / new quant format) costs +1 switch case per backend — O(1), not O(fusions).

## 2. Phased plan (with vt:: touch-points)

**Phase 0 — skeleton + ONE fusion end-to-end, CPU+CUDA (prove the seam).** Pick `kFusedAddRmsNorm` (residual-add + gemma-RMSNorm) as the first: real ≥2-primitive row-reduction chain, **no quant terminal** (dodges the backend-negotiated tail for the proof), and an existing golden — `RmsNorm(..., residual)` at `qwen3_5.cpp:2322`. Touch-points:
- NEW `include/vt/fused_recipe.h` (the POD: `Op`/`Reduce` enums, `Step`, `FusedRecipe`).
- NEW `include/vt/recipes.h` (`constexpr kFusedAddRmsNorm`).
- `include/vt/ops.h`: +1 `OpId::kFusedChain` (insert before `kCount`, line 59); +1 `FusedChainFn` alias (near line 264); +1 `void FusedChain(...)` decl.
- `src/vt/ops.cpp`: +1 `FusedChain` validate-then-dispatch wrapper (the `ops.cpp:35-46` pattern).
- `src/vt/cpu/cpu_ops.cpp`: register `kFusedChain` → Tier-0 composite + a scalar Tier-1 interpreter (register near `cpu_ops.cpp:823`).
- `src/vt/cuda/cuda_ops.cu`: register `kFusedChain` → Tier-1 interpreter kernel reusing the `AttnQkNormRopeGateKernel` block-reduce skeleton (`cuda_ops.cu:390`; register near `:621`).
- Test: golden parity **interpreter == composite == existing `RmsNorm(residual)`**, bit-identical, CPU+CUDA.

**Phase 1 — roll out the finite Class-A list, CUDA+CPU.** Transcribe into `recipes.h` and swap call-sites; retire each bespoke op once its recipe reaches parity: `kSiluMulFp4Quant` (`qwen3_5.cpp:2391`), `kRmsNormFp4Quant` (Task 11), `kQkNormRopeGate` (`:1615/:1758` — the biggest bespoke kernel; Tier-2 spec reuses `cuda_ops.cu:390`), `kMoeCombineGate` (`:2028/:2214`), GDN glue (`GdnPostConv/GdnGBeta/GdnConvSplit`). Each = 1 builder + 1 call-site swap; add Tier-2 spec **only where measured hot**. Record the `kFusedChain` runtime-POD as a `porting-inventory §9` deviation (vLLM's answer is Inductor codegen), with each recipe citing its upstream fusion-pass file.

**Phase 2 — portability proof, deferred to backend #2.** Port the Tier-1 interpreter ONCE to Vulkan/Metal → all recipes light up at Tier-1 speed; implement the quant terminal opcode in each backend's native format.

## 3. Honest payoff

- **Net-new CUDA prefill from this framework: low single digits.** Class-A is only ~5–8% of the 20% gap, and most of it is **already shipped**; the framework's CUDA contribution is (a) safely flipping the default-OFF fusions on behind a trustworthy CPU oracle, and (b) the few genuinely-unfused chains (GDN bf16 glue, full-attn f32 glue, memory-traffic gaps). It does **not** by itself close 20%.
- **The dominant residual stays hard and out of scope:** fp4-GEMM tactic/StreamK config, FA-2 cp.async + softmax-log2, GDN memset/tiling (Tasks 16/18/21/23) — Class B/C.
- **The real payoff is structural + strategic** (the user's top-weighted axes): retires the *growing* bespoke-kernel surface into ~150 lines of declarative recipes + one interpreter per backend; structurally eliminates CPU/CUDA parity drift; and makes backend #2 inherit **every** fusion at Tier-0 correctness free + Tier-1 speed for a single kernel port — instead of re-hand-writing N kernels × M backends (candidate #4's structural gap reopening on every backend).

## 4. Risks + decision points for you

1. **Tier-1 interpreter perf vs hand-fused** on ALU-heavy recipes (CUDA local-mem register file). **Do not retire a hand kernel until Tier-1 (or Tier-2) is measured at parity on the real bf16-intermediate chain.** → Decision: which recipes need Tier-2.
2. **Quant tail is not portable.** → Decision: accept the norm/rope/gate prefix as portable and the quant opcode as backend-negotiated (matches `backends.md`)?
3. **Scope fence.** → Decision: confirm Class B → cutlass-EVT separate track and Class C → own tracks; the recipe framework owns Class-A HBM-pass chains only, and is not sold as "the" 20% answer.
4. **Timing/ROI on CUDA (the sharpest call).** Since most Class-A gaps are already hand-shipped, the near-unanimous judge read is: **build the skeleton + retire 1–2 ops now** to lock the seam and the CPU oracle, but **defer full retirement + Tier-2 + the Vulkan/Metal port until either a default-OFF fusion needs safe gating or backend #2 actually starts.** → Decision: green-light Phase 0 + partial Phase 1 now; gate the rest on backend #2. Building the whole framework ahead of backend #2 is candidate #3's "premature generality" trap.

## Ranking
- [runtime-interpreter] overall=4 port=4 scale=4 — fatal: The three headline virtues (zero-codegen, hand-fused perf, full portability) cannot hold simultaneously for the perf-critical quant chains. Variant A is zero-codegen and portable but backs the runtime-indexed register file with CUDA local memory, so it will not match hand-fused on ALU-heavy/many-op recipes; recovering that perf forces variant B, which reintroduces the per-recipe build-time specialization/enumeration it claimed to eliminate. Simultaneously the terminal QUANT_BLK_FP4/FP8 opcodes are NVIDIA-specific -- backends.md states quant layout is backend-negotiated (Metal/Vulkan/XPU use GGUF k-quants, not NVFP4) -- so the norm/elementwise prefix ports but the tail of exactly the recipes that close the prefill gap (rmsnorm+fp4, silu+mul+fp4) does not port with one opcode. And the premise is oversized: the gap-scan's 20 confirmed prefill gaps are dominated by flashinfer fp4-GEMM autotune, FA-2 cp.async/softmax, and GDN kernel-internal tiling/memset -- all explicitly out of FusedChain's scope -- so the standalone HBM-pass chains it owns are a minority of the 20%.
- [buildtime-cpp-codegen] overall=3 port=3 scale=4 — fatal: The 'every backend at once' portability headline is false on the two HIGHEST-weighted future targets, and it is propped up by an inaccurate 'incremental' claim. (1) Vulkan/Metal are not reached by templates, so by the proposal's own admission they need a hand-written per-language text emitter that must re-implement each archetype's control flow — grid-stride loops AND block/subgroup reductions with shared memory + barriers — in GLSL/MSL. That is a second compiler backend, not a one-time cost that 'touches zero shader code per recipe'; reductions in particular are the hard, unproven part. (2) The GemmEpilogue archetype has NO shader realization path at all: CUTLASS/EVT is CUDA-only, so fused GEMM epilogues simply do not port to Vulkan/Metal — the design silently drops its most valuable fusion class on exactly the backends the user weights highest. (3) The load-bearing 'CUTLASS EVT is already vendored and DRIVEN by our fp8/nvfp4 GEMMs, so GemmEpilogue is incremental' claim is wrong: the cited path (cuda_matmul_fp8_cutlass.cu:163 CollectiveBuilder, :213-214 fusion_args.alpha_ptr; nvfp4 mirror) uses the DEFAULT LinearCombination epilogue with only a per-tensor alpha scalar — no EVT visitor tree exists in the tree today. Generic recipe-DAG -> Sm120 block-scaled EVT visitor lowering is new, brittle, compile-time-heavy engine work (EVT is CUTLASS's most template-error-prone, slowest-compiling surface, and sm120a block-scaled epilogues are constrained), not a catalog line — so the effort is materially higher and the perf on the epilogue half is unproven, not 'inherited'.
- [opgraph-ir-lowering] overall=3 port=4 scale=4 — fatal: Premature high-effort generality: the architecture spends its heaviest cost — a from-scratch recorder + shape-carrying symbolic Values + a DAG pattern-matcher (conceptually re-implementing Inductor's job) — to earn a payoff that is entirely DEFERRED and SPECULATIVE for this project's current state. Its portability/scalability value only materializes across backends that do not yet exist (Metal/Vulkan T2, ROCm T3) on a model set that is fixed at one model, while the actual stated GOAL (close the 27B prefill gap) is delivered solely by the CUDA/cutlass-locked EVT codegen path, which is orthogonal to and independent of the entire portable IR. The IR is thus separable from the only thing that pays: a skeptic ships EVT behind the 20 existing levers today and closes the gap, and the recorder/matcher machinery — plus its net-new silent-wrong-fusion attack surface (use-count/exclusivity predicate bugs) introduced into a golden-parity-disciplined codebase — buys nothing until a second real backend lands. Secondary: prefill is exactly the workload, prefill token counts are varlen and may not hit shape buckets, so the CUDA-graph replay win the perf case leans on is forfeited precisely on the target path, leaving per-step std::function interpreter dispatch that the design waves away as 'negligible' without measurement.
- [curated-vt-fusedops] overall=3 port=3 scale=3 — fatal: Perf-portability is N×M hand kernels and the design amortizes ZERO of it. The composite fallback gives a new backend CORRECTNESS for free but literally zero speed, so every future Vulkan/Metal/ROCm/XPU target must re-hand-write all ~12 fused kernels (plus re-derive each backend's native GEMM-epilogue mechanism) to close the SAME 20% prefill gap — the gap structurally reopens on every new backend, which is exactly the priority (portability of the actual goal, fast prefill) the user weighted highest. Compounding it: the linchpin of even the correctness-portability claim does not exist yet — GetOp (ops.cpp:26) VT_CHECK-throws on an unregistered (op,backend), so 'a brand-new backend inherits correctness for free' is FALSE until the composite fallback is built; today an unimplemented fusion op hard-crashes the new backend rather than degrading gracefully.
- [declarative-dsl-lazy] overall=3 port=3 scale=3 — fatal: The GEMM-epilogue chain — the candidate's own named single biggest 27B lever — cannot be materialized from a runtime heap-tree IR by a generic walker. cutlass EVT is compile-time template metaprogramming, and the repo today ships only the auto-builder LinearCombination/alpha_ptr epilogue (verified in cuda_matmul_{nvfp4,fp8}_cutlass.cu), with NO visitor trees. So "the IR maps near-mechanically onto EVT" is net-new work that resolves to either a finite pre-compiled specialization menu (= per-pattern hand kernels, which kills the scalability thesis) or runtime nvrtc/JIT codegen (per-backend, and explicitly disowned for the default path — plus Vulkan/Metal have no EVT analog, so it re-forks per backend anyway). The abstraction breaks exactly where the perf is. Compounding it: the gap-scan pattern set is finite and mostly already implemented as hand ops (SiluMulFp4Quant, AttnQkNormRopeGate), and the remaining 27B levers are non-fusion (memset elimination, cp.async, GEMM tactic steering, bf16 plumbing) that don't fit the IR at all — so the DSL's declarative-cheap-Nth-fusion amortization applies mainly to fusions the model doesn't need, and never repays its high build + per-fusion parity (f32 accumulation-order pinning) + net-new-surface ownership cost. It also sits off-mirror: vLLM has no runtime lazy-IR (its answer is Inductor codegen), making this a §9 deviation in a project whose discipline doc calls reinvention a defect.
