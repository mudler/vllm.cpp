# Marlin NVFP4 W4A16 grouped-MoE drop-in — feasibility investigation

Grounded feasibility basis for reusing vLLM's **Marlin W4A16 NVFP4** kernel (the
weight-only fp4×bf16 kernel vLLM actually selects for the 35B's MoE experts)
behind a `vt::` op, to close the ~5× residual speed gap on GB10 (sm_121) for the
35B — the analog of the cutlass fp4×fp4 win for the 27B. All vLLM cites are
`file:line` relative to the pinned checkout `/home/mudler/_git/vllm` @ `e24d1b24`.
Build+run probe run on `dgx.casa` (GB10, compute_cap 12.1, CUDA 13.0.88, nvcc)
2026-07-05.

**Verdict: GO — the key risk (Blackwell support) is resolved POSITIVELY.**
The Marlin NVFP4 W4A16 kernel — both the dense and the grouped/MoE variants —
compiles to **native `sm_121` SASS** (not PTX-JIT) with nvcc 13 `-arch=sm_121a`,
and the dense path **runs and is numerically correct on GB10**. Marlin was
originally an Ampere/Ada kernel, but vLLM's current version explicitly supports
the SM12x family, and its only tensor-core/async PTX (`mma.sync…m16n8k16` bf16 +
`cp.async`) is fully native on Blackwell GB10.

## 1. Where the kernel lives (cited)

vLLM keeps the live, torch-stable-ABI Marlin kernels under `csrc/libtorch_stable/`:

- **Dense W4A16**: `quantization/marlin/marlin.cu` — host entry
  `marlin_mm(...)` / the registered `marlin_gemm` op. NVFP4 is selected by
  `b_type == vllm::kFE2M1f` (fp4 weights) + `s_type == vllm::kFE4M3fn` (fp8
  block scales) (`marlin.cu:600,786,824`). The compute kernel is the templated
  `Marlin<...>` in `quantization/marlin/marlin_template.h` (2081 lines).
- **MoE grouped W4A16**: `moe/marlin_moe_wna16/ops.cu` — host entry
  **`moe_wna16_marlin_gemm(...)` (`:543`)**, registered `:897`. Same NVFP4
  selection (`ops.cu:603,780,817`). Compute kernel = `Marlin<...>` in
  `moe/marlin_moe_wna16/marlin_template.h` (2241 lines). **This is the 35B's
  actual MoE-expert kernel** (`ModelOptNvFp4FusedMoE(use_a16=True)` →
  `marlin_moe.py` → `moe_wna16_marlin_gemm`).
- **Kernel instantiations**: generated by `generate_kernels.py <arch>` (jinja2)
  into gitignored `sm80_kernel_*.cu`. The NVFP4 W4A16 config is
  `b_type=kFE2M1f, s_type=kFE4M3fn, a_type∈{kFloat16,kBFloat16}, c_type=a_type,
  group_blocks=1` (group size 16) (`generate_kernels.py:94-100`). For bf16
  activation it emits `sm80_kernel_bfloat16_fe2m1f_bfloat16.cu`.

**Marlin needs NO cutlass** — it is standalone CUDA + inline PTX (`cp.async` in
`marlin.cuh:98-166`, `mma.sync.aligned.m16n8k16` in `marlin_mma.h:38-79`). The
W4A16 math dequantizes fp4→bf16 in-register (`dequant.h`) and multiplies with
**bf16 tensor cores** (`mma…m16n8k16.row.col.f32.bf16.bf16.f32`), i.e. weight-
only. No fp8/fp4 tensor-core hardware is required, so it runs on any sm_80+.

### Blackwell support is explicit in-source (the load-bearing point)
`moe/marlin_moe_wna16/generate_kernels.py:16-18` (identical in the dense
generator):
```
# SM89 and the SM12x family (SM120 RTX 5090, SM121 DGX Spark GB10)
# fully support mma.sync.aligned.m16n8k32.row.col.f32.e4m3.e4m3.f32.
```
vLLM's Marlin generator names **SM121 DGX Spark GB10** as a first-class target.
The arch guards in the kernel only cap the *lower* bound (`__CUDA_ARCH__ < 750`
errors; `== 750` uses the Turing m16n8k8 path; `< 800` disables cp.async) —
**there is no upper cap** that would exclude sm_120/121. Blackwell takes the
standard sm_80+ path.

## 2. Dependency posture — torch-free-liftable? YES (same shape as the cutlass lift)

The **compute core is pure CUDA**. `marlin_template.h`, `marlin.cuh`,
`marlin_dtypes.cuh`, `dequant.h`, `marlin_mma.h`, `kernel.h` include only CUDA
headers + `core/scalar_type.hpp`. `scalar_type.hpp`'s only torch touch is
`STD_TORCH_CHECK` via `<torch/headeronly/util/Exception.h>` (a header-only shim,
trivially replaceable with a `throw`) — the same POD ScalarType the cutlass lift
already coexists with.

The **torch coupling is confined to the two host launchers** (`marlin.cu`,
`marlin_moe_wna16/ops.cu`) via `torch::stable::Tensor` — `.data_ptr()`,
`.size()`, `.scalar_type()`, `torch::stable::empty` (workspace), `DeviceGuard`,
`get_current_cuda_stream`. Every one maps 1:1 onto `vt::Tensor` + our stream,
exactly as done for `cuda_matmul_nvfp4_cutlass.cu`. No separate vendored lib —
the whole thing is ~8.2k lines of self-contained csrc (core headers 3.4k +
launchers/repack 2.4k + generated instantiations).

## 3. Build + run probe (LOAD-BEARING) — PASS

Staged the marlin csrc onto GB10, generated the sm12.1 kernels
(`python generate_kernels.py 12.1`), compiled the **NVFP4 W4A16** instantiation
with `nvcc 13.0.88 -std=c++17 -arch=sm_121a -O2 --expt-relaxed-constexpr`:

| Kernel (= the vLLM path) | Compile | Native SASS | HMMA / LDGSTS |
|---|---|---|---|
| `sm80_kernel_bfloat16_fe2m1f_bfloat16.cu` — **dense** W4A16 | OK, 16.7 s | `arch=sm_121a` | — |
| `sm80_kernel_bfloat16_fe2m1f_bfloat16.cu` — **MoE grouped** W4A16 | OK, 24.7 s | `arch=sm_121a` | 2736 / 1703 |

`cuobjdump -sass` confirms **real sm_121 machine code** (`HMMA` bf16 tensor-core
+ `LDGSTS` cp.async), i.e. it is *not* PTX-only kept for JIT. This resolves the
primary open question — Marlin's Ampere-era PTX assembles to Blackwell SASS.

**Run + correctness (dense, on GB10 via vLLM's own repack/apply):** quantized a
random bf16 `[512,1024]` weight to NVFP4 (`scaled_fp4_quant`, linear layout),
repacked with `gptq_marlin_repack(num_bits=4)` + `nvfp4_marlin_process_scales`,
ran `marlin_gemm` (`b_q_type=float4_e2m1f`) vs the full-precision matmul:
```
out [64,512] bf16 finite=True
cosine_sim(marlin, fullprec) = 0.99548   rel_err = 0.0951   → PASS
```
(the 9.5% is just fp4 weight-quant error on random data, not a kernel bug.)
The installed vLLM 0.24.0 wheel on the box (built for torch 2.11+cu130, cap
12.1) ships `marlin_gemm` **and** `moe_wna16_marlin_gemm` — i.e. vLLM runs this
exact grouped-MoE kernel in production on this GB10 class.

## 4. The repack requirement (Marlin's interleaved layout)

Marlin needs weights pre-permuted into its tile-interleaved layout at load time.
vLLM's path (mirror it in a load-time repack, CPU or GPU):

- **Weight**: `gptq_marlin_repack(b_q_weight, perm=∅, size_k, size_n,
  num_bits=4, is_a_8bit=false)` (`quantization/marlin/gptq_marlin_repack.cu:285`)
  — input is the modelopt `[N, K/2]` packed fp4 viewed as int32 `[K/2/4, N]`ᵀ,
  padded to Marlin's `padded_n/padded_k` (see
  `marlin_utils_fp4.prepare_fp4_layer_for_marlin` /
  `prepare_nvfp4_moe_layer_for_marlin`).
- **Block scales**: fp8-e4m3 `[N, K/16]` → transpose, pad, `marlin_permute_scales`,
  then `nvfp4_marlin_process_scales` (folds a scale_factor out of the global
  scale so the kernel's fp16 accum stays in range) →
  `nvfp4_marlin_process_global_scale`.
- **MoE**: `gptq_marlin_moe_repack` does all E experts at once
  (`awq_marlin_repack.cu` sibling); `prepare_nvfp4_moe_layer_for_marlin`
  (`marlin_utils_fp4.py:309`) is the per-expert w13/w2 pad+repack+scale recipe.

The repack is done ONCE at load; the hot path is only `moe_wna16_marlin_gemm`.

## 5. Lift plan (STEP 1-3, actionable — mirror the cutlass drop-in)

1. **Vendor** the marlin core headers + the two launchers into `src/vt/cuda/`
   (own TUs, `-arch=sm_12{0,1}a`, like `cuda_matmul_nvfp4_cutlass.cu`); generate
   the single NVFP4-bf16 instantiation (or a curated subset of thread configs).
   Swap `torch::stable::Tensor` → `vt::Tensor`; `STD_TORCH_CHECK` → `VT_CHECK`.
2. **New vt op** `vt::MoeGroupedGemmNvfp4Marlin` mirroring `moe_wna16_marlin_gemm`
   (`ops.cu:543`) — the grouped signature (sorted_token_ids, expert_ids,
   num_tokens_past_padded, topk_weights, moe_block_size, top_k). Reuse the
   existing MoE align/sort scaffolding (`kMoeRouterTopK`, moe_align).
3. **Repack at load**: lift `gptq_marlin_repack` + the scale/global-scale
   permute into the 35B weight loader (`qwen3_5_weights.cpp`), producing the
   Marlin-layout fp4-resident buffers alongside (or replacing) the current
   packed buffers. Gate the whole thing `VT_NVFP4_MARLIN=1`.
4. **Wire** it at the `MoeGroupedGemmNvfp4` call site in the 35B forward
   (`src/vt/cuda/cuda_matmul_nvfp4.cu` grouped path / `qwen3_5.cpp`), replacing
   the hand-written `MoeGroupedGemmNvfp4Wmma` (~15% of peak) when the gate is on.
5. **Correctness FIRST**: unit (Marlin out == reference dequant-matmul, fp4 tol —
   dense already validated here at cos 0.995) → **35B `test_qwen36_paged_engine`
   16/16 token-for-token** → 27B unaffected. THEN A/B measure (35B total/output/
   TTFT/TPOT before→after; report the Marlin MoE-GEMM TFLOPS vs the ~15%-of-peak
   hand-written kernel) vs `vllm bench throughput --input-len 1024 --output-len
   128 --num-prompts 8`. **Merge only on 16/16 + a measured win.**

REUSED vs new: reuse the entire MoE routing/align/silu scaffolding, the fp4
weight-resident load path, and the cutlass-style TU/gate structure; new = the
two lifted launchers, the NVFP4 kernel instantiation, and the load-time repack.

## 6. Remaining risk / effort

- The lift itself is the large work item (~5k lines of templated kernel +
  launcher + the load-time repack + MoE align plumbing + 16/16 parity + A/B).
  Compile risk is retired; the residual risk is (a) getting the repack + scale
  layout bit-exact (the fiddly part — but vLLM's `marlin_utils_fp4` is the exact
  recipe to mirror) and (b) whether Marlin's MoE-GEMM actually beats the tuned
  hand-written wmma kernel at the 35B's decode shapes (M small, per-expert
  K=2048→I=512). Marlin is heavily tuned for exactly this M-small W4A16 regime,
  so the expectation is positive, but it must be *measured*, not assumed.
- Probe artifacts on the box: `~/work/marlin-probe/` (staged csrc + generated
  kernels + compiled `.o`s); dense run harness `/tmp/marlin_nvfp4_run.py`.

## 7. STEP 1-2 DONE + kernel proven BIT-EXACT (branch `feature/marlin-nvfp4`)

**2026-07-05, GB10 sm_121a, CUDA 13.0.** Executed the lift:

- **Vendored** the torch-free Marlin slice into `src/vt/cuda/marlin/` (see

  the kernels we compile.
- **`vt::MoeGroupedGemmNvfp4Marlin`** (`src/vt/cuda/cuda_moe_marlin.cu`) +
  OpId/`MoeMarlinArgs`/Fn/dispatcher in `ops.{h,cpp}`. Mirrors the NVFP4 branch
  of `moe_wna16_marlin_gemm` (no act-order/zp/bias/8-bit-act branches; group
  size 16, bf16 act/out, `use_fp32_reduce`).
- **CMake gate `VLLM_CPP_MARLIN`** (sm_12xa). Critical flag:
  `-static-global-template-stub=false` — nvcc 13 gives `__global__` template
  instantiations *static* linkage by default, so the `Marlin<>` addresses taken
  in the dispatcher's `kernel_selector.h` are **undefined at link** across TUs
  (reproduced on GB10). With the flag, cross-TU linkage resolves.

**Build**: the full in-tree CUDA build (`-DVLLM_CPP_CUDA=ON
-DVLLM_CPP_CUDA_ARCHITECTURES=121a`) is GREEN with Marlin ON — `libvllm` + all
test executables link (cross-TU `Marlin<>` symbols resolve in-tree). 0 errors.

**Correctness (unit)**: `tools/marlin/` — vLLM's own repack
(`prepare_nvfp4_moe_layer_for_marlin` recipe) + `moe_wna16_marlin_gemm` golden
vs OUR vendored `marlin_mm` on the IDENTICAL repacked inputs:
