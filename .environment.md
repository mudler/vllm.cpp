## 2026-07-10 driver upgrade and UVM crash mitigation

The upgraded host runs Linux 6.18.38 with NVIDIA open driver 595.71.05. The
prior crash path was traced to persistent caches allocated as pageable
`std::vector` storage and then exposed as CUDA tensors. This RTX reports
`cudaDevAttrPageableMemoryAccess=1` but `cudaDevAttrIntegrated=0`; pageable
access means HMM/UVM can service a host pointer, not that the discrete GPU has
physically unified memory.

The CUDA backend now requires both attributes before reporting unified memory.
On this discrete GPU, full-attention KV, GDN SSM, and convolution caches are
allocated with the backend allocator and verified as `cudaMemoryTypeDevice`.
The shared temporary pool is keyed by backend and supports an optional
`VT_POOL_MAX_CACHED_MB` ceiling with allocation/eviction statistics.

Progressive validation completed without an Xid, oops, soft lockup, UVM
lazy-free call, or PMA eviction:

- focused CUDA backend/cache ownership tests;
- 31/31 GDN Triton-toggle tests and the Qwen dense forward tests;
- 224 prompts at concurrency 56, 1024 input / 128 output, completing at
  5,554.19 total tok/s with launch blocking enabled;
- native Nsight traces of both engines and three unprofiled repetitions each.

The old trace contained hundreds of thousands of Unified-memory migrations.
The fixed traces contain H2D/D2D/D2H copies and no Unified transfer category.
Four read-only kprobes in `tools/bench/watch_nvidia_uvm_pressure.bt` attached
successfully and observed no UVM pressure event during the experiments. The
kernel emitted four isolated `refcntRequestReference_IMPL` state warnings during
profiler/NVML startup, but no Xid or execution failure; the GPU returned to
146-166 MiB idle after every run.

## 2026-07-10 Qwen3.5-4B local comparison

Workload: 128 deterministic ShareGPT rows, exactly 1,024 input tokens and 128
generated tokens per request, concurrency 32, BF16 model, max model length
4,096, scheduler token budget 2,048, seed 0. The project used the sm_120
RelWithDebInfo Triton-AOT build with all three GDN AOT toggles enabled and a
1,024 MiB pool ceiling; vLLM 0.24.0 used its production compile/CUDA-graph path.
Both sides used temperature 1 and ignored EOS.

| Engine | Total tok/s (3 runs) | Mean | Output tok/s mean | Ratio |
| --- | --- | ---: | ---: | ---: |
| vLLM | 6,639.71 / 6,685.60 / 6,682.52 | 6,669.28 | 741.03 | 1.000x |
| vllm.cpp Triton AOT, parallel sampler | 5,617.17 / 5,612.36 / 5,613.22 | 5,614.25 | 623.81 | 0.842x |

This is not an accepted parity result because the separate greedy precondition
fails: 9/16 complete sequences match vLLM on two deterministic prompt corpora.
Disabling all three GDN AOT kernels improves one corpus to 10/16; AOT and hand
paths match each other on 13/16. The AOT path therefore contributes some
numerical drift but does not explain the whole BF16-model gap.

The original matched trace identified `RandomSampleKernel` at 81.7% of GPU
time: 109.98 seconds over 527 calls, or 208.7 ms/call, because one thread per
request scanned the 248K-token vocabulary. The replacement mirrors vLLM's
parallel execution structure: up to 256 blocks produce per-row partial winners
and a final reduction selects the token, while retaining the project's
documented SplitMix RNG. `VT_FAST_RANDOM_SAMPLE=0` restores the scalar kernel
for same-binary checks.

The new trace reports `RandomSamplePartialKernel` at 1.5% of GPU time (377.82
ms total over 527 calls, 0.717 ms/call), softmax at 0.7%, and the final reduction
below 0.01%. A same-binary scalar run produced 1,087.09 tok/s, so the default
parallel path is a 5.16x end-to-end improvement and now runs within 2.4% of the
project's greedy throughput. The remaining 15.8% same-sampling gap to vLLM is
in the model/runtime path, not sampling. Pool peak remains 954.87 MiB with zero
evictions and no Unified-memory transfers.

Before the sampler replacement, the same vllm.cpp workload was repeated with
temperature 0, which bypasses `RandomSampleKernel` and uses the parallel greedy argmax:
5,753.30 / 5,754.45 / 5,757.01 total tok/s (mean 5,754.92) and mean 639.44
output tok/s. This is a 5.30x improvement over the temperature-1 project run,
but only 0.862x of the recorded vLLM temperature-1 mean. That cross-mode ratio
is not an accepted apples-to-apples result; it shows that removing the sampler
bottleneck would leave a model/runtime gap. The subsequent same-sampling run
measures that open gap at 15.8% against a fresh vLLM denominator.
