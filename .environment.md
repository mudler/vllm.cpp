## Local implications for vllm.cpp

- This machine cannot use the original remote DGX/Spark gate assumptions.
- The 16 GB VRAM budget is too small for the recorded full gate-model
  workloads. The latest recorded project memory numbers for the gate models are
  well above this machine's VRAM budget.
- The repo currently has full-engine support for the Qwen3.6 gate-family paths
  and synthetic tiny test engines. Generic small dense models such as Qwen3-0.6B
  are referenced in the historical record but are not currently a supported
  full-engine benchmark target.
- A local performance plan should therefore first make a small real model work
  in both vLLM and this project, then benchmark that same model on the same
  workload. Gemma 4 family small models are a candidate to evaluate next, but the
  repo needs model support before they can be meaningful vllm.cpp benchmarks.

- PyTorch: 2.11.0+cu130

- Compute capability reported by PyTorch: `(12, 0)`

Extra Nix shell fixes needed for Python/JIT packages:

- `libstdc++.so.6` from `gcc14.cc.lib` must be on `LD_LIBRARY_PATH`.
- `TRITON_LIBCUDA_PATH=/run/opengl-driver/lib` avoids Triton's hard-coded
  `/sbin/ldconfig -p` probe, which does not exist on NixOS.
- cuRAND headers/libraries are needed for FlashInfer sampling JIT
  (`curand.h`); the flake exports CUDA `dev` and `include` outputs through
  `CPATH`, `C_INCLUDE_PATH`, and `CPLUS_INCLUDE_PATH`.
- Local cache directories used for this run:
  `.hf-cache`, `.vllm-cache`, `.torchinductor-cache`, `.triton-cache`,
  `.flashinfer-cache`.

Common vLLM benchmark environment:

```sh
HF_HOME=$PWD/.hf-cache \
XDG_CACHE_HOME=$PWD/.vllm-cache \
TORCHINDUCTOR_CACHE_DIR=$PWD/.torchinductor-cache \
TRITON_CACHE_DIR=$PWD/.triton-cache \
TRITON_LIBCUDA_PATH=/run/opengl-driver/lib \
FLASHINFER_WORKSPACE_BASE=$PWD/.flashinfer-cache \
CUDA_VISIBLE_DEVICES=0
```

Model choice:

- `google/gemma-4-E4B-it` was tried first because it is the largest small
  Gemma 4 target that would be useful if it fit. It does not fit in bf16 on
  this 16 GB GPU: model initialization OOMs after about 14.38 GiB process GPU
  memory, with another local process using about 604 MiB and Xorg using about
  385 MiB.
- `google/gemma-4-E4B-it-qat-mobile-transformers` has an official Gemma QAT
  config, but vLLM 0.24 rejects `quant_method: gemma`; it is not a usable vLLM
  oracle target here.
- `google/gemma-4-E2B-it` is the largest Gemma 4 target that ran cleanly in
  vLLM bf16 on this machine. The text stack is still non-trivial: 35 layers,
  hidden size 1536, 8 attention heads, 1 KV head, head dim 256, sliding window
  512, and max position 131072.

vLLM reports for Gemma 4 E2B:

- Architecture: `Gemma4ForConditionalGeneration`
- Attention: Gemma 4 heterogeneous head dimensions force
  `AttentionBackendEnum.TRITON_ATTN`; FA4 is not used.
- Chunked prefill: enabled, `max_num_batched_tokens=8192`.
- Model load memory: about 9.9 GiB.
- At `--max-num-seqs 32`, `--max-model-len 4096`,
  `--gpu-memory-utilization 0.88`, available KV cache was about 2.57 GiB /
  148,660 tokens, with vLLM reporting 36.29x maximum concurrency at 4096
  tokens/request.

Interpretation:

- For the 1024/128 synthetic workload, 112 is the largest tested useful
  parallel setting at the current memory utilization; 128 initializes but OOMs
  on activation allocation.
- For the 2048/128 synthetic workload, 64 is the largest tested useful setting;
  80 initializes but OOMs on activation allocation.
- Throughput runs still log two Triton JIT warnings during inference
  (`_compute_slot_mapping_kernel`, `kernel_unified_attention`), so these are
  practical local oracle numbers, not polished final reproducibility numbers.
- vllm.cpp cannot yet run this Gemma 4 model, so no apples-to-apples vllm.cpp
  comparison exists yet.
