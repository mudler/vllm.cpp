# M0.5 Tokenizer Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Byte-level BPE tokenizer (HF `tokenizer.json` + GGUF vocab sources) with incremental detokenization — token-for-token identical to HF `tokenizers` on a diverse corpus.

**Architecture:** Scope is exactly the gate models' tokenizer family: GPT-2-style byte-level BPE with the Qwen split pattern (Llama-3's variant differs only in its regex — same machinery). Components: generated Unicode tables → hand-written pretokenize scanner (no regex engine) → merge-ranked BPE → special-token longest-match pre-pass. Detokenizer mirrors upstream `vllm/v1/engine/detokenizer.py` incremental semantics. Verification is oracle-driven: HF `tokenizers` (in the dgx venv) dumps corpus goldens; C++ replays them in CI. **Anything unsupported in a tokenizer.json (non-BPE model, unknown pre_tokenizer) throws loudly — no silent wrong tokenization.**

**Tech Stack:** nlohmann/json, generated C++ tables (Python `unicodedata` at gen time only), doctest.

## Global Constraints

- Token-for-token parity with HF tokenizers is the bar; the corpus must cover: ASCII prose, code (mixed indentation, symbols), CJK, emoji (incl. ZWJ sequences), Arabic/Hebrew (RTL), leading/trailing/multiple spaces, newline runs, contractions ('s 're 'll), digits (1/2/3/4+ runs), special tokens embedded mid-text, and the empty string.
- Mirrored placement: `src/vllm/tokenizer/` (upstream keeps tokenizers in Python/Rust — header comment `// vllm.cpp original (tokenizer); semantics mirror HF tokenizers byte-level BPE.`); detokenizer at `src/vllm/v1/engine/detokenizer.{h,cpp}` with header `// Ported from: vllm/v1/engine/detokenizer.py @ e24d1b24`.
- Generated files are committed with their generator + regeneration command in the file header.
- Untrusted-input discipline for tokenizer.json/GGUF vocab (counts bounded, ids validated < vocab size).
- Warnings-as-errors; TDD; push to `main` authorized.

### Task 1: Unicode tables (generated)

**Files:** `tools/gen_unicode_data.py`; generated `src/vllm/tokenizer/unicode_data.{h,cpp}`; test `tests/vllm/test_unicode_data.cpp`.

**Interfaces:** `namespace vllm::tok { enum class UCat : uint8_t { kOther, kLetter, kNumber, kSeparator, kMark, kPunct, kSymbol, kControl }; UCat Category(uint32_t cp); bool IsWhitespace(uint32_t cp); /* python str.isspace semantics per HF */ }` — implemented as sorted (start,end,cat) range tables + binary search.

Generator: Python stdlib `unicodedata` (the venv/oracle NOT needed — system python3 on the dev box is fine; record `unicodedata.unidata_version` in the generated header). Map general categories: L* → kLetter, N* → kNumber, Z* → kSeparator, M* → kMark, P* → kPunct, S* → kSymbol, C* → kControl. Whitespace: `str.isspace()` plus 0x85/0xA0 handling exactly as Python. Emit ranges (merge adjacent), expect ~4k ranges. Also UTF-8 encode/decode helpers `DecodeUtf8(std::string_view, size_t& pos) -> uint32_t` (invalid bytes → U+FFFD, advance 1 — HF byte-level never sees invalid UTF-8 in practice because input is UTF-8 text, but be defensive) and `EncodeUtf8(uint32_t, std::string&)`.

Tests: spot-check categories ('a' letter, '駅' letter, '5' number, '½' number(No), ' ' separator+whitespace, '\t' control+whitespace, '🙂' symbol, combining accent mark), UTF-8 round-trip incl. 4-byte cp, invalid-byte handling. Commit `feat(tokenizer): generated unicode tables + utf8 helpers`.

### Task 2: Pretokenize scanner (Qwen + Llama-3 patterns)

**Files:** `src/vllm/tokenizer/pretokenizer.{h,cpp}`; test `tests/vllm/test_pretokenizer.cpp`.

**Interfaces:** `enum class SplitPattern { kQwen2, kLlama3 }; std::vector<std::pair<size_t,size_t>> Pretokenize(std::string_view text, SplitPattern p);` — byte-offset spans, concatenation reconstructs input exactly.

Both models use minor variants of the GPT-4/cl100k pattern. Authoritative reference: the `pre_tokenizer` regex strings inside each model's tokenizer.json (fetch from the DGX checkpoints — Qwen3.6 27B snapshot has tokenizer.json; record both regexes verbatim in a comment). Qwen2 pattern (verify against the file):
`(?i:'s|'t|'re|'ve|'m|'ll|'d)|[^\r\n\p{L}\p{N}]?\p{L}+|\p{N}|(?# 1 digit)| ?[^\s\p{L}\p{N}]+[\r\n]*|\s*[\r\n]+|\s+(?!\S)|\s+`
Implement as a hand-written scanner with explicit rules in priority order (contraction | optional-nonletter+letters | single digit (Qwen splits digits individually!) | optional-space+punct-run+newlines | newline-runs | whitespace-before-nonspace (keep last space attached to next) | whitespace). Each rule documented with the regex fragment it implements. Llama-3 variant: digits grouped 1-3 (`\p{N}{1,3}`), case-insensitive contractions, otherwise same shape — implement both behind the enum.

Tests (golden spans written by hand from known HF behavior — verified later by Task 6 corpus): `"Hello world"` → ["Hello", " world"]; `"I'm  fine\n\n"`; `"x123"` → Qwen ["x","1","2","3"] vs Llama3 ["x","123"]; `" 你好"`; `"a...b"`; tabs+spaces mix; trailing spaces. Reconstruction property test over random ASCII+UTF-8 strings. Commit `feat(tokenizer): byte-level pretokenize scanner (qwen2 + llama3 patterns)`.

### Task 3: BPE core + tokenizer.json loading

**Files:** `src/vllm/tokenizer/{bpe,tokenizer}.{h,cpp}`; test `tests/vllm/test_bpe.cpp`.

**Interfaces:**
```cpp
namespace vllm::tok {
struct SpecialToken { std::string text; int32_t id; bool special; };
class Tokenizer {
 public:
  static Tokenizer FromHfJson(const std::string& tokenizer_json_path);   // throws on non-BPE/unknown pre_tokenizer
  static Tokenizer FromGguf(const GgufFile& f);                          // Task 4
  std::vector<int32_t> Encode(std::string_view text) const;              // no BOS/EOS added (caller policy)
  std::string Decode(const std::vector<int32_t>& ids) const;             // full decode (non-incremental)
  const std::string& TokenText(int32_t id) const;                        // raw token string (byte-level alphabet)
  int32_t VocabSize() const; int32_t EosId() const; int32_t BosId() const;  // -1 when absent
  SplitPattern Pattern() const;
};
}
```
Mechanics (all in bpe.cpp, unit-tested separately): GPT-2 `bytes_to_unicode` bijection (committed table constructed in code: printable ranges 0x21-0x7E, 0xA1-0xAC, 0xAE-0xFF map to themselves; others map to 0x100+n in order); merge table `unordered_map<pair-of-token-strings → rank>` (hash on concatenated with separator); per-pretoken BPE: start from byte-mapped unicode chars, repeatedly merge the lowest-rank adjacent pair (linked-list + heap or O(n²) scan — n is tiny; O(n²) scan is fine and simpler), then map merged symbols → ids via vocab. Special/added tokens: longest-match scan of the raw text BEFORE pretokenization (matching HF added_tokens behavior; only exact matches split the text). Loading: parse model.type=="BPE" (else throw), vocab map, merges (both "a b" strings and [a,b] array forms — newer format), added_tokens (id, content, special flag), detect pattern: if the pre_tokenizer JSON contains `\p{N}{1,3}` → kLlama3 else kQwen2 (also accept an explicit Split regex equal to one of the two recorded patterns; anything else → throw with the regex in the message). eos/bos from post_processor if trivially extractable else -1 (callers use config's eos_token_id — record note).

Tests: bytes_to_unicode bijection (256 distinct, round-trip); tiny handcrafted vocab/merges (e.g. vocab {a,b,c,ab,abc...}) exercising merge order; added-token splitting mid-text; unknown-model-type throws. Commit `feat(tokenizer): BPE core + tokenizer.json loader`.
