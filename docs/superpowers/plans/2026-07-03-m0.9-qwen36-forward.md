# M0.9 Qwen3.6 Forward + Registry — the M0 Exit Criterion

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Assemble the Qwen3.6 model forward (embed → 40 × decoder → norm → lm_head → logits) from the vt ops built in M0.2/M0.6/M0.7/M0.8 plus a new dense causal-attention op, loading real 35B/27B weights (NVFP4 experts dequant-to-bf16 + bf16 dense), and prove **logits parity + token-for-token greedy decode** vs the pinned oracle. This closes M0.

**Architecture:** Single-sequence, non-paged, correctness-grade (no engine, no KV paging — that is M1). Mirrored placement: `src/vllm/model_executor/models/{registry,qwen3_5}.{h,cpp}` (semantics from pinned `models/qwen3_next.py`+`qwen3_5.py`), weight mapping in the loader. The forward is an explicit function over vt ops (no module system), per the design doc. The 1-in-4 full-attention layers need a new **dense causal attention** op (qk-norm + partial RoPE + GQA + output gate) — paged attention stays M1.6.

**Oracle strategy (the crux — Task 1 de-risks it, RE-SCOPED after Task 1):**
there is NO tiny hybrid model (gate models are 27B/35B; Qwen3-0.6B is plain-dense,
wrong arch). Task 1 established (see `.agents/specs/qwen36-forward-notes.md`) that the
two gate checkpoints are DIFFERENT architectures: **35B =
`Qwen3_5MoeForConditionalGeneration`** (MoE, 256e/top-8, hidden 2048, 40 layers,
**modelopt W4A16** weight-only 4-bit → bf16 activations) and **27B =
`Qwen3_5ForConditionalGeneration`** (DENSE, hidden 5120, 64 layers,
**compressed-tensors W4A4** — weights AND activations 4-bit, NO experts). So the
**35B is the PRIMARY M0-exit gate**: it exercises the MoE forward end-to-end and
its weight-only-4bit→bf16 scheme matches our `DequantNvfp4ToBf16 + bf16 matmul`
path cleanly, so it can be held to a logit-atol bar plus token-for-token greedy.
The **27B is a SECONDARY dense-path check**: it never touches the MoE block, and
its W4A4 oracle (quantized activations, real cutlass FP4 GEMM) diverges from our
bf16-dequant path by MORE than bf16 rounding, so its bar is greedy-token-match
ONLY, not logit-atol. **Oracle engine = pip vLLM 0.24.0** (the pinned overlay is
blocked — §3 of the notes; every forward-math module is byte-identical to the pin,
only weight-loader plumbing + ROCm differ). **Per-layer goldens** (one GDN decoder
layer + one full-attn decoder layer + embed + final-norm + lm_head) are dumped for
BOTH checkpoints (`_35b` MoE, `_27b` dense) to keep iteration cheap; the full-model
end-to-end logits+greedy gate is the **35B**.
