# M1.5 InputBatch + BlockTable (MRV2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** The persistent batch + step-input build — `BlockTable` (per-request block-id tensors + slot mapping), `InputBatch` (the long-lived per-slot state updated incrementally from `SchedulerOutput`: add new / apply cached diffs / swap-remove finished), and the step-input construction (`query_start_loc`, `seq_lens`, `positions`, `slot_mapping`, logits indices) — ported 1:1 from vLLM V1's **Model Runner V2** (`vllm/v1/worker/gpu/{input_batch,block_table}.py`) so the model runner (M0.9 forward, later the batched runner) can consume a scheduled step. Behavioral (no CUDA/model in the tests — the tensors are plain host arrays here; device placement is the runner's concern), CI-testable.

**Architecture:** Mirrored `src/vllm/v1/worker/gpu/{input_batch,block_table}.{h,cpp}` (← `vllm/v1/worker/gpu/`). This is MRV2 (the project's committed direction; the scheduler already emits the MRV2 shape with `prefill_token_ids` + resumed-as-new). The InputBatch is a **persistent** structure: it holds per-slot arrays (token ids, positions, block tables, sampling params, num_computed_tokens) mutated in place across steps, NOT rebuilt each step — this is the vLLM performance design. Scope T0: the single-sequence + small-batch path the gate models need; use plain host `std::vector`/arrays for the "tensors" (the vt-device version is a later perf concern). Defer LoRA/spec-decode/multimodal/structured-output slot state behind marked stubs.

### Task 2: InputBatch (persistent) — add / remove / condense
Read `/home/mudler/_git/vllm/vllm/v1/worker/gpu_input_batch.py` (**V1**, NOT `gpu/input_batch.py`) — the `InputBatch`/`CachedRequestState`. **Source-of-truth is the V1 `gpu_input_batch.py`**, because (a) the V1 `InputBatch` holds the `MultiGroupBlockTable` Task 1 ported, and (b) it matches this plan's own description of a *persistent* structure that holds the block table and is mutated in place (add/remove/condense). The MRV2 `gpu/input_batch.py` is a **TRANSIENT** per-step object (it holds NO block table; the persistent state is split into `req_states` + `BlockTables` on the runner) — that staged design is deferred to M2 (see the "Deferred to M2 (perf)" note below); do NOT port it or `gpu/states.py` for the block-table-holding InputBatch at T0.

Port the persistent structure: per-slot arrays (req_ids indexed by slot, token_ids_cpu, num_computed_tokens_cpu, num_prompt_tokens, block_table via MultiGroupBlockTable, sampling metadata), `add_request(new_req_data, slot)`, `remove_request(req_id) → slot`, `condense()` (swap-remove to keep dense), `num_reqs`, the req_id↔index maps. Match upstream's incremental-mutation contract. Defer LoRA/spec/mm/structured slot state (marked). Unit tests ported (add fills a slot, remove + condense keeps dense + fixes indices, num_reqs).

**CONTRACT ADAPTER — consume the MRV2 scheduler-output shape, not MRV1:** `add_request`/cached-update take the M1.4 MRV2 `SchedulerOutput` shape — new reqs carry `prefill_token_ids`; resumed reqs are folded in as new; cached diffs = `num_computed_tokens` + `new_block_ids`. Do NOT port the V1 runner's MRV1-shape admission handling (`resumed_from_preemption` / `resumed_req_ids` / per-req `all_token_ids` reconstruction) — that path is **dead** under our MRV2 scheduler. Port the V1 InputBatch *algorithm* (the persistent array bookkeeping) but drive it from the MRV2 contract.

**Deferred to M2 (perf):** the MRV2 `gpu/` staged storage — `StagedWriteTensor`/`UvaBackedTensor` `BlockTables`, the fused-Triton `apply_staged_writes`/`gather_block_tables`/`compute_slot_mappings`, and the transient-`InputBatch` + `req_states`/`block_tables`-on-runner decomposition — is the device-perf upgrade, ported when M2 introduces the `vt` device.
