# M1.5 InputBatch + BlockTable (MRV2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** The persistent batch + step-input build ‚Äî `BlockTable` (per-request block-id tensors + slot mapping), `InputBatch` (the long-lived per-slot state updated incrementally from `SchedulerOutput`: add new / apply cached diffs / swap-remove finished), and the step-input construction (`query_start_loc`, `seq_lens`, `positions`, `slot_mapping`, logits indices) ‚Äî ported 1:1 from vLLM V1's **Model Runner V2** (`vllm/v1/worker/gpu/{input_batch,block_table}.py`) so the model runner (M0.9 forward, later the batched runner) can consume a scheduled step. Behavioral (no CUDA/model in the tests ‚Äî the tensors are plain host arrays here; device placement is the runner's concern), CI-testable.

**Architecture:** Mirrored `src/vllm/v1/worker/gpu/{input_batch,block_table}.{h,cpp}` (‚Üê `vllm/v1/worker/gpu/`). This is MRV2 (the project's committed direction; the scheduler already emits the MRV2 shape with `prefill_token_ids` + resumed-as-new). The InputBatch is a **persistent** structure: it holds per-slot arrays (token ids, positions, block tables, sampling params, num_computed_tokens) mutated in place across steps, NOT rebuilt each step ‚Äî this is the vLLM performance design. Scope T0: the single-sequence + small-batch path the gate models need; use plain host `std::vector`/arrays for the "tensors" (the vt-device version is a later perf concern). Defer LoRA/spec-decode/multimodal/structured-output slot state behind marked stubs.
