# M0.10 GGUF model load (k-quant dequant + Qwen3.6 loader)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Load a GGUF Qwen3.6 checkpoint into the model — the user-mandated "can we
read gguf" gate. The GGUF READER (metadata + tensor infos + block geometry) already
exists (`gguf_reader.{h,cpp}`, `4aeae62`). M0.10 adds: (1) the **dequant** (GGUF quant
blocks → f32/bf16) for the types the APEX Qwen3.6 GGUFs use, and (2) a **GGUF →
Qwen3_5MoeWeights loader** (tensor-name mapping + config metadata → HfConfig), wired
into the model-load path so `model_loader` accepts a `.gguf` file. Validated on the
real APEX GGUFs on dgx (dgx-pending, like the 35B safetensors gate); the dequant math
is unit-tested on CPU.

**APEX quant types (from .agents/gguf-nvfp4-notes.md §3 — histogrammed on dgx):**
F32(0), Q8_0(8), Q3_K(11), Q4_K(12), Q5_K(13), Q6_K(14), IQ2_S(22), IQ4_XS(23). The
Compact/Balanced variants are pure K-quant + Q8_0; IQ2_S/IQ4_XS only in Mini/Quality.
NO NVFP4 in APEX (but the reader supports id 40 for the separate NVFP4 GGUFs).

## Global Constraints

- The dequant block formats are ggml's (mainline). Match ggml's dequantize_row_* /
  the block layout EXACTLY (block elems + byte layout per type — the geometry is
  already in gguf_reader's GgmlTraits). Reference: llama.cpp
  /home/mudler/_git/llama.cpp `ggml/src/ggml-quants.c` (dequantize_row_q4_K etc.) +
  `ggml-common.h` (block_q4_K etc. struct layouts). A wrong bit-unpack → garbage weights.
- The loader mirrors `qwen3_5_weights.{h,cpp}` (the safetensors loader) — same OwnedTensor
  bf16 targets, same transposes/layout the M0.9 forward expects; only the SOURCE
  (GGUF k-quant tensors) + the tensor NAMES (llama.cpp gguf qwen naming, e.g.
  `blk.N.attn_q.weight`, `token_embd.weight`, `output.weight`, `blk.N.ffn_*_exps.weight`)
  differ. Map GGUF names → our weight struct.
- The GGUF metadata → HfConfig (n_layers, n_heads, n_kv_heads, hidden, rope_theta,
  expert count, etc. — the `qwen3_5moe`/`qwen35moe` arch keys). Mirror how llama.cpp
  reads them.
- Warnings-as-errors; TDD; push authorized (CPU dequant tests; the real-GGUF load on dgx).

### Task 1: K-quant + legacy dequant (F32, Q8_0, Q4_K, Q5_K, Q6_K, Q3_K, Q4_0)
Read llama.cpp `ggml/src/ggml-quants.c` (dequantize_row_q8_0/q4_K/q5_K/q6_K/q3_K/q4_0)
+ `ggml-common.h` (the block_qN_K struct layouts: the super-block of 256 = 8×32, the
6-bit scales/mins, the quantized nibbles). Port `src/vllm/model_executor/model_loader/
gguf_dequant.{h,cpp}`: `DequantGgufToF32(ggml_type, const uint8_t* data, int64_t numel)
-> std::vector<float>` (+ a bf16 variant) dispatching per type:
- F32(0): memcpy. Q8_0(8): d(f16) × int8. Q4_0(2): d × (nibble-8).
- Q4_K(12), Q5_K(13), Q6_K(14), Q3_K(11): the super-block K-quant dequant — the 6-bit
  block scales/mins (d, dmin f16 + the packed 6-bit scales) × the per-32 quantized
  values. Match ggml's exact bit layout (the scales packing is the tricky part — port
  it byte-for-byte).
Unit tests: dequant a synthetic block of each type whose bytes you construct from known
(scale, quant) values → assert the output f32 matches the hand-computed ggml formula.
Cross-check block byte sizes against GgmlTraits (already in the reader). A round-trip
(quantize a known vector with the ggml formula → dequant → close) OR a golden block
(a real ggml-produced block + its expected f32, if obtainable). Correctness-grade.
