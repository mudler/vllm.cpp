# M1.8 EngineCore + step loop + batched paged runner

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Close M1 — wire the ported pieces (M1.4 Scheduler + M1.5 InputBatch/
prepare_inputs + M1.6 paged attention/GDN metadata + M0.9 Qwen3.6 forward + M1.7
Sampler) into a **running V1 engine loop**: `EngineCore.step()` (schedule →
execute_model → sample → update_from_output), the single-process/single-GPU
executor seam, the InputProcessor (text prompt → EngineCoreRequest) and
OutputProcessor (EngineCoreOutputs → RequestOutput, incremental detokenize +
string-stop), and — the central piece — the **batched paged model runner** whose
forward consumes the paged KV cache instead of M0.9's dense single-sequence
attention. Scope T0: the correctness-grade text-completion path the gate models +
the serving MVP need. CPU-testable; the real 35B greedy parity re-runs through the
paged path on dgx.

**Architecture (the V1-algorithm contract; MRV2 staged storage deferred to M2 —
see `.agents/vllm-v1-v2.md`):** We target the MRV2 runner's CONTRACT/SHAPE
(`vllm/v1/worker/gpu/model_runner.py` — asserts `prefill_token_ids`, splits
`execute_model` forward from `sample_tokens`) but IMPLEMENT it via the V1 host-array
algorithm we already ported (M1.5 persistent `InputBatch` holds the
`MultiGroupBlockTable`; `prepare_inputs` emits the dense-order `StepInputs`; M1.7
`SamplingMetadata` built fresh each step in that SAME dense `[0, num_reqs)` order).
**This composes with no `idx_mapping` slot-indirection** — attention metadata (M1.6),
the gathered logits (`hidden_states[logits_indices]`), the `SamplingMetadata` rows,
and the sampled-token write-back all align on ONE dense order. The MRV2 staged
per-slot GPU buffers (`RequestState`/`InputBuffers`/`UvaBackedTensor`,
`Sampler.add_request`/`apply_staged_writes`, `idx_mapping`) + the persistent-slot
`gpu/sample/sampler.py` are **axis-2 storage, deferred to M2**. The executor collapses
to a **direct pass-through** over the runner (no `collective_rpc`/`WorkerWrapperBase`/
`Future`/multiproc/Ray/DP). Mirrored files: `src/vllm/v1/engine/{core,input_processor,
output_processor}.{h,cpp}`, `src/vllm/v1/executor/executor.{h,cpp}`, the batched runner
in `src/vllm/v1/worker/gpu/runner.{h,cpp}`.

## Global Constraints

- 1:1 with pinned e24d1b24 — VERIFY the pinned CURRENT API before porting. Same
  class/method/field names. Ported files carry `// Ported from: <path> @ e24d1b24`.
- **THE CENTRAL REFACTOR — forward dense→paged.** M0.9's `Qwen3_5Model::Forward`
  (src/vllm/model_executor/models/qwen3_5.cpp) is single-sequence DENSE:
  `FullAttnBlock` calls `vt::Attention` (no KV cache/block table, line ~300);
  `GdnBlock` calls `vt::GdnPrefill` with a fresh per-call state (line ~229). M1.8
  rewires it to the **batched paged path**: full-attn layers call
  `vt::ReshapeAndCache` (write new K/V into the paged NHD cache) + `vt::PagedAttention`
  (read K/V from the cache via block_table/seq_lens); GDN layers use the batched
  `GDNAttentionMetadata` (M1.6) + persistent mamba-state, prefill/decode-segmented.
  The forward now takes attention metadata + KV caches + GDN metadata, not a bare
  (token_ids, positions). The single-sequence M0.9 greedy result MUST be reproduced by
  a batch-of-1 full-prefill + single-token decodes through the paged path — this is the
  M0-exit-equivalent correctness gate for the batched forward.
- **THE FOUR-WAY ORDERING CONTRACT (biggest integration risk — de-risk FIRST).** The
  runner reorders the batch **decode-first-then-prefill**
  (`reorder_batch_to_split_decodes_and_prefills`) before building metadata, because the
  GDN split (`split_decodes_and_prefills`, M1.6) assumes decodes lead. ALL of {attention
  metadata seq_lens/block_tables, GDN segmentation, the forward's input_ids/positions,
  the gathered logits_indices, the SamplingMetadata rows, the sampled-token write-back}
  must be built in that ONE order. A mismatch yields correct shapes with SILENTLY WRONG
  results (no crash). Before wiring sample+write-back, add an **identity assertion test**:
  for a batch of {1 decode req, 1 prefill req}, verify logits_indices / the sampling row
  / the attention seq_lens+block_table row / the write-back slot all resolve to the SAME
  request. (Inert for the batch-of-1 gate + pure-decode/pure-prefill batches; only bites
  a MIXED hybrid batch.)
- **GDN-STATE ZEROING (M1.6 caller obligation — CLOSE HERE).** Our vt::GdnPrefill/
  GdnDecode read the mamba `state` buffer UNCONDITIONALLY. The batched GDN-layer assembly
  MUST gather `state[prefill_state_indices]` then zero the rows where
  `prefill_has_initial_state==0` before calling vt::GdnPrefill (mirrors
  qwen_gdn_linear_attn.py:1512-1513). Wire this in the forward refactor + test a
  fresh-vs-continuing mixed GDN batch.
- Warnings-as-errors; TDD; push authorized (CPU CI; the 35B paged greedy gate on dgx).

### Task 3: Batched paged forward (THE central refactor)
Read the pinned `qwen3_next.py` forward + `qwen_gdn_linear_attn.py` (how the attention/
GDN layers consume the attention backend + mamba state in the batched/paged case) +
re-read `.agents/qwen36-forward-notes.md`. Refactor `Qwen3_5Model::Forward`
(src/vllm/model_executor/models/qwen3_5.{h,cpp}) to the **batched paged path**:
- New signature: `Forward(token_ids, positions, attn_metadata (per-KV-group:
  CommonAttentionMetadata + GDNAttentionMetadata), kv_caches (the (num_blocks,2,
  block_size,H,D) full-attn buffers + the mamba-state buffers), ...) -> logits-ready
  hidden states`. Positions/token_ids remain the flattened dense-order step inputs.
- **Full-attn block:** replace `vt::Attention` with `vt::ReshapeAndCache(new K/V →
  paged NHD cache at slot_mapping)` then `vt::PagedAttention(out, q, k_cache, v_cache,
  block_table, seq_lens, query_start_loc, scale, causal)`. Keep the gemma-RMSNorm +
  partial-NeoX RoPE + output-gate exactly as M0.9. K/V slices are the strided unbind(1)
  views (M1.6 Task 2 contract).
- **GDN block:** drive `vt::GdnPrefill`/`vt::GdnDecode` from the batched
  `GDNAttentionMetadata` (num_prefills/num_decodes segmentation, prefill_query_start_loc,
  prefill_state_indices, prefill_has_initial_state); use the PERSISTENT mamba-state
  buffer (indexed by state_indices), not a fresh per-call state. **Wire the GDN-state
  zeroing:** gather state rows + zero where prefill_has_initial_state==0 before
  GdnPrefill (the M1.6 caller obligation). Conv-state likewise persistent.
- **Validation (the gate):** a batch-of-1 full-prefill of the M0.9 prompt then 16
  single-token greedy decodes THROUGH THE PAGED PATH must reproduce M0.9's token-for-
  token result ("…Germany is Berlin.\nThe capital of France is Paris, and the"). Also a
  small CPU test: paged full-prefill of one sequence == M0.9 dense forward on the same
  input (per-layer + logits). GDN: a fresh-vs-continuing state test (a decode step reading
  persisted state == the prefill+decode run in one shot). CPU green; 35B paged greedy on
  dgx is the milestone gate (dgx-pending note).

### Task 4: Batched runner (execute_model + sample + write-back) + KV alloc + ordering gate
Read `vllm/v1/worker/gpu/model_runner.py` (`execute_model`/`sample_tokens`/`sample`/
`postprocess_sampled`, the T0 slice) + how KV cache tensors are allocated from
`KVCacheConfig`. Port `src/vllm/v1/worker/gpu/runner.{h,cpp}`:
- **KV cache allocation** from the M1.3 `KVCacheConfig`: the per-full-attn-layer
  `(num_blocks,2,block_size,H,D)` buffers + the per-GDN-group mamba-state + conv-state
  buffers. (inventory §3 "KV tensor allocation from KVCacheConfig" T0.)
- **The decode-first reorder** (`reorder_batch_to_split_decodes_and_prefills`) applied to
  the InputBatch before metadata build, so all consumers see one order.
- `execute_model(scheduler_output)`: update_states (M1.5) → **reorder** → prepare_inputs
  (M1.5 StepInputs) → build attention metadata (M1.6 MakeCommonAttentionMetadata + the
  GDN metadata builder, per KV group) → forward (Task 3, with the KV caches) → stash
  hidden states.
- `sample_tokens()`: `hidden_states[logits_indices]` → `compute_logits` (lm_head) →
  build `SamplingMetadata` from the InputBatch (M1.7 make_sampling_metadata, dense order)
  → `Sampler.forward` (M1.7) → `ModelRunnerOutput` (sampled_token_ids per req, logprobs)
  → **write sampled tokens back** into the InputBatch/requests (append_output_token_ids)
  so the next step's prepare_inputs sees them.
- **THE ORDERING IDENTITY TEST (de-risk gate, per Global Constraints):** a batch of
  {1 decode, 1 prefill} — assert logits_indices / sampling row / attention seq_lens+
  block_table row / write-back slot all resolve to the same request.
- Defer (marked): cudagraphs, spec-decode/rejection, LoRA, mm/encoder, PP, EPLB,
  kv-connector, prompt-logprobs, async/AsyncOutput (return ModelRunnerOutput synchronously),

  decode over N steps (KV cache grows, tokens append); 2-req batch step; the ordering
  identity test; KV allocation shape from a fake KVCacheConfig.
