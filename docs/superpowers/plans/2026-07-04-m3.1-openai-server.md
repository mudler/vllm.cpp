# M3.1 OpenAI server (completions + chat, streaming) + chat templates

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** The OpenAI-compatible serving layer over the M1.8 `LLMEngine` — the
`/v1/completions` and `/v1/chat/completions` endpoints (SSE streaming +
non-streaming), `/v1/models`, `/health`, the OpenAI request/response/error
shapes, and the request→SamplingParams mapping — plus the chat-template
application (M3.2, folded in) needed for chat. Scope T0: text completion + chat
completion, greedy + sampled, streaming deltas + full responses, on CPU over the
working engine. Ported 1:1 in shape from vLLM's `vllm/entrypoints/openai/`.
Tool-calling (M3.3) and grammars (M3.4) layer on top later.

**Architecture:** The OpenAI-protocol logic (request parsing → SamplingParams,
generate loop → response/SSE-chunk formatting) is **decoupled from the HTTP
transport** so it is unit-testable without a socket. Mirrored:
`src/vllm/entrypoints/openai/{protocol,serving_completion,serving_chat,
serving_models}.{h,cpp}` (← `vllm/entrypoints/openai/`), the chat template in
`src/vllm/entrypoints/chat_template.{h,cpp}`, and the HTTP wiring +
`examples/server` last. **HTTP-layer decision:** vendor a header-only HTTP lib
(cpp-httplib — the same choice as llama.cpp's server; MIT, single header, no
external deps) under `third_party/`; the route handlers call the decoupled
serving logic. Record this as a dependency deviation (it is a transport lib, not
a compute/ML dep — consistent with the no-pytorch/no-ggml rule).

## Global Constraints

- 1:1 in SHAPE with pinned e24d1b24 — the OpenAI protocol field names, the
  request→params mapping, the response/chunk JSON shapes, the error shapes must
  match `vllm/entrypoints/openai/protocol.py` + `serving_{completion,chat}.py`
  so clients (OpenAI SDK, LocalAI) work unchanged. Ported files carry
  `// Ported from: vllm/entrypoints/openai/<file> @ e24d1b24`.
- JSON via the already-vendored nlohmann/json (used elsewhere in the tree).
- The serving logic drives the M1.8 `LLMEngine` (add_request/step/generate) — the
  streaming path consumes the OutputProcessor's DELTA RequestOutputs and formats
  SSE `data:` chunks; non-streaming accumulates to the final CompletionOutput.
- Warnings-as-errors; TDD; push authorized (CPU CI). The HTTP server itself is
  smoke-tested; the protocol/formatting logic is unit-tested without a socket.

### Task 1: OpenAI protocol types + request→SamplingParams mapping
Read `vllm/entrypoints/openai/protocol.py` (CompletionRequest/CompletionResponse/
CompletionResponseChoice/CompletionStreamResponse; ChatCompletionRequest/Response/
StreamResponse + ChatMessage/DeltaMessage; UsageInfo; ErrorResponse; the
`to_sampling_params()` mapping). Port `src/vllm/entrypoints/openai/protocol.{h,cpp}`
— the T0 field subset with nlohmann/json (de)serialization: model, prompt/messages,
max_tokens, temperature, top_p, top_k, n, stop, stream, seed, presence/frequency/
repetition_penalty, logprobs, echo (defer), the response/chunk/usage/error shapes.
The `to_sampling_params()` → our SamplingParams (map each field 1:1; run PostInit).
Defer (marked): logit_bias/tools/tool_choice/response_format/structured (M3.3/M3.4),
best_of, prompt_embeds, mm. Unit tests: a JSON CompletionRequest round-trips + maps
to the right SamplingParams; a ChatCompletionRequest parses messages; error shape.

### Task 2: S/serving_completion + serving_chat (decoupled from HTTP)
Read `vllm/entrypoints/openai/serving_completion.py` + `serving_chat.py` +
`serving_engine.py` (the generate + format loop). Port
`src/vllm/entrypoints/openai/{serving_completion,serving_chat}.{h,cpp}` — a handler
holding an `LLMEngine&` that: parses the request → SamplingParams, calls the engine
(streaming: yield SSE `CompletionStreamResponse`/`ChatCompletionStreamResponse`
chunks per DELTA RequestOutput, ending with `data: [DONE]`; non-streaming: the full
`CompletionResponse`/`ChatCompletionResponse` with usage). Match the exact chunk
cadence (role delta first for chat, then content deltas, then finish_reason).
Return a stream of formatted JSON strings + a final, so the HTTP layer (Task 4)
just writes them. Defer tool/grammar/logprob-payload paths (marked). Unit tests
(NO socket): drive the completion handler with a request over a small synthetic
engine (reuse the M1.8 test harness), collect the streamed chunks, assert they
concatenate to the non-streaming response text + the finish_reason + usage counts;
the chat handler with a 2-message conversation.

### Task 3: Chat template application (minja-subset, M3.2 folded in)
Read how vLLM applies chat templates (`vllm/entrypoints/chat_utils.py` +
`transformers` apply_chat_template; the tokenizer_config.json `chat_template`
Jinja). Port a **minja-subset** Jinja renderer (`src/vllm/entrypoints/chat_template.{h,cpp}`)
sufficient for the Qwen3.6 chat template (the constructs it uses: `for` over
messages, `if` role checks, `{{ }}` interpolation, `add_generation_prompt`,
the bos/eos + `<|im_start|>`/`<|im_end|>` special tokens) → the prompt string the
completion path tokenizes. Load the template from tokenizer_config.json. Defer the
full Jinja spec (only the subset the gate-model templates need; mark unsupported
constructs as a loud error). Unit tests: the Qwen3.6 template applied to a
[system,user] conversation produces the exact expected prompt string (compare to a
pinned transformers `apply_chat_template` dump for the same messages).

### Task 4: HTTP server (vendored cpp-httplib) + routes + example CLI
Vendor cpp-httplib under `third_party/httplib.h`. Port
`src/vllm/entrypoints/openai/api_server.{h,cpp}` — the routes `/v1/completions`,
`/v1/chat/completions` (SSE when stream=true via chunked `text/event-stream`),
`/v1/models`, `/health`, `/version` — each parsing the request, calling Task 2's
handlers, writing the response/SSE stream + the OpenAI error shape on failure.
`examples/server/main.cpp` — a CLI that loads a model (weights + tokenizer + KV
config → LLMEngine) and serves. Smoke test: start the server on a port, curl
`/health` + a non-streaming `/v1/completions` + a streaming one, assert the OpenAI
response shape. (This is the llama.cpp-style server gate.) Keep the model-load
path minimal (safetensors — GGUF is M0.10); a tiny synthetic model for the smoke
test if real weights aren't present on the CI box.

## Self-review notes
- The protocol shapes + the request→params mapping + the SSE chunk cadence are the
  client-compat core — match upstream's JSON exactly so OpenAI-SDK/LocalAI clients
  work unchanged.
- Decouple serving logic from HTTP (Task 2 testable without a socket); the HTTP
  layer (Task 4) is thin.
- Chat template = minja-subset for the gate models only; loud-error on unsupported
  Jinja rather than silently wrong prompts.
- T0 = completions + chat, streaming + non-streaming, greedy + sampled, safetensors
  model load; tools (M3.3), grammars (M3.4), GGUF (M0.10), C-API (M3.5) are separate.
