# M3.3 Tool / function calling

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** OpenAI tool/function calling over the M3.1 chat server: `tools` +
`tool_choice` in the chat request, the tools rendered into the prompt via the chat
template, tool-call **parsing** from the model output (Qwen + Hermes formats), the
`tool_calls` response/delta shapes (non-streaming + streaming), and **grammar-forced
JSON** for `tool_choice=required`/named (via M3.4). This is an explicit MVP feature.

**Architecture:** Mirror `vllm/entrypoints/openai/tool_parsers/` (the per-format
parsers) + the `chat_completion/serving.py` tool path. Files:
`src/vllm/entrypoints/openai/tool_parsers/{abstract,hermes,qwen3}.{h,cpp}`, the
protocol tool fields (protocol.h), the serving_chat tool wiring, the chat-template
tools injection (M3.2 engine already supports the needed Jinja; the Qwen template's
tool branch uses constructs the minja engine may need — extend if required).

### Task 1: Tool protocol types + request wiring
Read `vllm/entrypoints/openai/protocol.py` (ChatCompletionToolsParam {type:"function",
function{name, description, parameters:schema}}, ToolChoice {none|auto|required|
{type:function, function{name}}}, ChatCompletionMessageToolCall / ToolCall {id,
type, function{name, arguments}}, FunctionCall, DeltaToolCall / DeltaFunctionCall).
Port into include/vllm/entrypoints/openai/protocol.h: the `tools` (list) +
`tool_choice` fields on ChatCompletionRequest; the ToolCall/FunctionCall in
ChatMessage/response choice; the DeltaToolCall in the stream delta. JSON (de)ser.
Unit tests: a request with tools + tool_choice parses; the tool_calls response +
delta shapes serialize to the exact OpenAI JSON.

### Task 2: Tool-call parsers (Hermes + Qwen3) — non-streaming
Read `vllm/entrypoints/openai/tool_parsers/{abstract_tool_parser,hermes_tool_parser,
qwen3xml_tool_parser or the qwen tool parser}.py`. Port the ToolParser ABC +
`extract_tool_calls(model_output, request) -> {tools_called, tool_calls[], content}`:
detect the `<tool_call>{"name":...,"arguments":{...}}</tool_call>` blocks (Hermes/Qwen
format), parse each into a ToolCall (generate an id, name, arguments-as-JSON-string),
and the remaining text as content. Match the exact detection + the multiple-tool-call
case. Unit tests: a model output with one/two `<tool_call>` blocks → the right
ToolCalls + content; no tool call → tools_called=false + the text as content;
malformed → graceful (content passthrough).

### Task 3: Streaming tool-call deltas + serving_chat tool wiring
Read the parser's `extract_tool_calls_streaming` + serving_chat.py tool path. Port
the incremental streaming parse (as tokens arrive, emit DeltaToolCall chunks: the
tool index + id + function.name once, then function.arguments deltas) and wire the
whole tool flow into src/vllm/entrypoints/openai/serving_chat.cpp: when the request
has tools, select the parser (by model/config — Qwen for the gate model), run
extract_tool_calls (non-stream) / the streaming parser (stream), and shape the
response with tool_calls + finish_reason="tool_calls". The chat template renders the
tools into the prompt (extend the M3.2 minja engine if the Qwen tool template branch
needs unsupported constructs — check + extend). Unit tests: a streamed tool call →
the delta sequence (name-first then arg deltas); non-stream tool call → the response
with tool_calls + finish_reason; content-only → normal.

### Task 4: Grammar-forced JSON for tool_choice=required / named
Read how vLLM forces JSON for tool_choice. When tool_choice=required or a named
function: build a grammar (via M3.4 JSON-schema→GBNF) from the tool's parameters
schema (for named: that tool's schema; for required: a JSON matching any of the
tools — an anyOf/alternation, or the simpler "one of the tool schemas") and set it as
the request's structured_outputs.json so the decode is CONSTRAINED to emit a valid
tool call. Wire it in serving_chat (before add_request). Unit tests: tool_choice=named
→ the structured_outputs grammar forces a JSON matching that tool's params (over the
synthetic engine, the constrained decode emits schema-valid JSON); tool_choice=required
→ constrained to a tool call. (If anyOf isn't supported by the JSON-schema→GBNF (M3.4
throws on anyOf), handle required by constraining to the single/first tool or extend
M3.4 minimally — note the choice.)
