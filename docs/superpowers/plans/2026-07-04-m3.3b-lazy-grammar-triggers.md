# M3.3b Lazy grammar triggers (relaxed tool_choice=auto)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Make `tool_choice=auto` RELAXED — the model may reply in plain text OR emit
a tool call; the tool-call JSON is grammar-constrained ONLY after the model chooses to
start one (emits the `<tool_call>` trigger). This is llama.cpp's "lazy grammar / grammar
triggers" mechanism (`src/llama-grammar.cpp` awaiting_trigger + `common/sampling.cpp`) —
and equals vLLM's xgrammar **StructuralTag** (which we'd deferred). `tool_choice=required`
/named stays forced-from-token-0 (M3.3 `caa6fa4`); `none` = no grammar.

**Why:** our structured-output backend currently masks EVERY step from token 0
(force-or-nothing). Forcing `<tool_call>` on `auto` would wrongly force a tool call when
the model wants to just reply. Doing nothing on `auto` leaves the tool-call JSON
unconstrained. The fix is a per-request **lazy / awaiting-trigger** state: inert (all
tokens allowed) until a trigger substring/token appears in the decoded output, then
activate + replay the buffered trigger tokens so the following JSON is constrained.

**Architecture:** Add the lazy-trigger state to the NATIVE grammar backend + thread a
"lazy + triggers" concept through StructuredOutputsParams → the manager → the
scheduler accept_tokens advance. Reference: llama.cpp `src/llama-grammar.cpp:1339-1427`
(awaiting_trigger no-op mask + trigger detect + replay), `common/common.h:132-143`
(trigger types), `common/sampling.cpp:208-260` (lazy vs non-lazy + trigger→regex),
`common/chat-auto-parser-generator.cpp:76-111` (tool_choice→lazy/triggers). vLLM parity
anchor: xgrammar StructuralTag (`vllm/tool_parsers/structural_tag_registry.py`).

## Global Constraints

- The lazy mechanism is the reusable core (port it, not llama.cpp's PEG/auto-parser).
  Trigger types (T0 subset): WORD (literal substring, our `<tool_call>`) + TOKEN (id);
  PATTERN (regex) marked-deferred unless trivial.
- **The invariant still holds:** while `awaiting_trigger`, fill_bitmask = ALL allowed
  (no constraint); once triggered, fill_bitmask==accept_tokens as before (M3.4 guard).
- Keep the M3.4 differential-test discipline: the fill==accept invariant + a
  lazy-specific test (awaiting → all allowed; post-trigger → constrained; replay
  advances the grammar so the JSON after `<tool_call>` is schema-valid).
- Warnings-as-errors; TDD; CLEAN rebuild before trusting green (stale-binary gotcha).

### Task 1: Lazy-trigger state in the native grammar (backend_native)
Read llama.cpp `src/llama-grammar.cpp:1339-1427` + `.h:139-148`. Add to `NativeGrammar`
(backend_native.{h,cpp}) a lazy mode: fields `bool lazy_`, `bool awaiting_trigger_`
(init = lazy_), `std::string trigger_buffer_`, `std::vector<std::string> trigger_words_`,
`std::vector<int32_t> trigger_tokens_`, and the tokenizer piece-decoding for the buffer.
- `fill_bitmask`: if `awaiting_trigger_` → set the WHOLE row allowed (all bits set) and
  return (mirror `apply_impl:1342` no-op). Else the existing DFS.
- `accept_tokens`: while `awaiting_trigger_`: if the token id is in trigger_tokens_ →
  flip awaiting_trigger_=false + accept that token into the FSM; else append the token's
  decoded piece to trigger_buffer_, search each trigger_word_ (a literal substring find)
  in the buffer; on a match at offset `start` → flip awaiting_trigger_=false and REPLAY
  the buffered bytes from `start` into the FSM byte-by-byte (mirror :1401-1421 — content
  before `start` was free text, discarded; the trigger `<tool_call>` and the JSON after
  advance the grammar). Return true (a token accepted while awaiting is always OK — free
  text). Once triggered, the normal byte-advance applies.
- `is_terminated`/`rollback`/`reset`/`validate_tokens`: account for the lazy state
  (reset → awaiting_trigger_=lazy_; rollback across the trigger boundary restores
  awaiting_trigger_ + the buffer — snapshot it).
- The backend `compile_grammar`: a way to build a LAZY grammar with trigger words. Add
  an overload / a LazyGrammarSpec so serving can request "grammar G, lazy, trigger
  words [W]". (The grammar G is the WRAPPED tool-call grammar from `caa6fa4`
  WrapSchemaAsToolCallGbnf; the trigger word is `<tool_call>`.)
Unit tests: a lazy grammar over `<tool_call>...</tool_call>` schema — before the trigger,
fill_bitmask allows ALL tokens (free text: a letter, a `{`, anything); feed free text
then `<tool_call>` → after the trigger the grammar constrains (only schema-valid JSON
bytes allowed); the JSON after `<tool_call>` is schema-constrained; feed text that never
triggers → stays all-allowed (a plain reply is unconstrained); replay correctness (the
`<tool_call>` prefix advanced the grammar so the `{` after it is required).

### Task 2: Thread lazy + triggers through the structured-output types + manager
Add to `StructuredOutputsParams` (sampling_params.h) a lazy/trigger carrier for the
grammar path (e.g. `bool structured_outputs.grammar_lazy` + `std::vector<std::string>
grammar_triggers`) — mirror common_chat_params.grammar_lazy/grammar_triggers
(common/chat.h:269-283). Thread it to the manager's grammar_init → the backend builds a
lazy grammar. The manager's grammar_bitmask already fills per-request rows via
grammar.fill_bitmask — no change needed (the grammar itself returns all-allowed while
awaiting). The scheduler accept_tokens advance (M3.4 Task 2) already calls
grammar.accept_tokens with the sampled tokens — that now drives the trigger detection.
Verify the whole flow: a lazy request → the runner masks nothing until `<tool_call>`,
then constrains. Unit tests: manager grammar_init with a lazy spec → the grammar is
lazy; grammar_bitmask over a lazy-awaiting request → all-allowed row; after accept_tokens
sees `<tool_call>` → constrained row.

### Task 3: serving_chat tool_choice=auto → lazy tool grammar
In serving_chat.cpp ApplyToolChoiceStructuredOutput: for `tool_choice=auto` (or unset,
the default) WITH tools present → set the wrapped tool-call grammar (the SAME
WrapSchemaAsToolCallGbnf over the anyOf of tools, `caa6fa4`) BUT LAZY with trigger word
`<tool_call>` (grammar_lazy=true, grammar_triggers=["<tool_call>"]). For required/named →
non-lazy (unchanged, forced). For none / no tools → nothing. Map to
common/chat-auto-parser-generator.cpp:76-111. The parser (Task 2/3) then extracts the
tool call from the (now possibly-present) `<tool_call>` wrapper; if the model replied in
plain text (no trigger fired), extract_tool_calls finds no `<tool_call>` → normal content
+ finish_reason="stop" (correct). Unit tests (over the synthetic engine / the shaping
functions): tool_choice=auto + tools → the request carries a LAZY grammar with the
`<tool_call>` trigger (not forced); a plain-text output → normal message (no tool_calls,
finish_reason="stop"); a `<tool_call>{...}</tool_call>` output → tool_calls +
finish_reason="tool_calls". Confirm auto no longer forces.
