# M3.3b Lazy grammar triggers (relaxed tool_choice=auto)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Make `tool_choice=auto` RELAXED â€” the model may reply in plain text OR emit
a tool call; the tool-call JSON is grammar-constrained ONLY after the model chooses to
start one (emits the `<tool_call>` trigger). This is llama.cpp's "lazy grammar / grammar
triggers" mechanism (`src/llama-grammar.cpp` awaiting_trigger + `common/sampling.cpp`) â€”
and equals vLLM's xgrammar **StructuralTag** (which we'd deferred). `tool_choice=required`
/named stays forced-from-token-0 (M3.3 `caa6fa4`); `none` = no grammar.

**Why:** our structured-output backend currently masks EVERY step from token 0
(force-or-nothing). Forcing `<tool_call>` on `auto` would wrongly force a tool call when
the model wants to just reply. Doing nothing on `auto` leaves the tool-call JSON
unconstrained. The fix is a per-request **lazy / awaiting-trigger** state: inert (all
tokens allowed) until a trigger substring/token appears in the decoded output, then
activate + replay the buffered trigger tokens so the following JSON is constrained.

**Architecture (RECONCILED â€” vLLM seam + llama.cpp mechanism):** Both vLLM and
llama.cpp confirm the design. **The 1:1 PARITY SEAM is vLLM's `STRUCTURAL_TAG`** (which
M3.4 Task 1 deferred): a `structural_tag` param on `StructuredOutputsParams`
(sampling_params.py:83), keyed `(STRUCTURAL_TAG, spec)` (request.py:96), compiled via
`compile_grammar(STRUCTURAL_TAG, spec)` (backend_xgrammar.py:96-110). vLLM's
integration (`fill_bitmask`/`accept_tokens` in the manager/scheduler/runner) is
**grammar-agnostic** â€” the lazy/triggered behavior lives entirely INSIDE the matcher
(xgrammar). So we port that seam 1:1 and implement the **lazy matcher NATIVELY** using
llama.cpp's mechanism (`src/llama-grammar.cpp:1339-1427`: awaiting_trigger no-op mask +
trigger detect + replay). The structural-tag SPEC (begin/content-schema/end + triggers)
mirrors vLLM's `StructuralTag`/`TagFormat`/`TriggeredTagsFormat`
(`vllm/tool_parsers/structural_tag_registry.py:213-269`) but is our own JSON (xgrammar's
exact bytes are xgrammar-internal â€” the spec content is backend-private; the SEAM is what
must match). **tool_choice â†’ structural tag** (structural_tag_registry.py:237-269):
auto â†’ `TriggeredTagsFormat` (lazy, triggers=[`<tool_call>`]); required â†’
`TagsWithSeparatorFormat(at_least_one)`; named â†’ `+stop_after_first`; none â†’ none.
**Design nuance:** vLLM bakes the tool name into the `begin` literal
(`<tool_call>\n{"name": "foo", "arguments": `, 2 surface variants/tool); we trigger on
`<tool_call>` alone then constrain `name` to a const-alternation via the grammar
(llama.cpp style) â€” equivalent accepted language, simpler state. Accept BOTH
`<tool_call>\n{...}` and `<tool_call>{...}` forms (vLLM's two variants). This SUBSUMES
`caa6fa4` (its WrapSchemaAsToolCallGbnf forced-wrapper = the required/named non-lazy
structural tag).
