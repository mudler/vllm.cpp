# M3.4 Grammars / structured output (native backend behind the 1:1 seam)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Constrained/structured decoding — JSON-schema, `json_object`, regex,
choice, and GBNF/EBNF grammars — that force the sampler to only emit grammar-valid
tokens (the per-step token bitmask → logits masked to -inf), exposed via the OpenAI
`response_format` + the sampling-params structured fields. This is an explicit MVP
feature ("grammars like llama.cpp") and the dependency for M3.3 forced-JSON tool
calling.

**ARCHITECTURE DECISION (from scoping — read carefully):** vLLM cleanly separates a
**backend-agnostic integration layer** (StructuredOutputManager + scheduler
`get_grammar_bitmask` + `GrammarOutput` + the runner/sampler `apply_grammar_bitmask`
+ the `StructuredOutputBackend`/`StructuredOutputGrammar` ABCs) from a **pluggable
grammar engine** (upstream ships 4: xgrammar/guidance/outlines/lm-format-enforcer).
We **port the integration layer 1:1** (name-for-name — this is the parity-critical
surface future PRs touch) and ship a **from-scratch NATIVE backend** (GBNF/EBNF +
JSON-schema→GBNF + regex/choice) behind the ported `StructuredOutputBackend` seam at
T0. **Vendoring xgrammar is a LATER milestone** (a second backend behind the same
proven seam, differentially parity-tested against the native one). This is NOT a
parity deviation — it mirrors upstream's own multi-backend design; the native
backend is a §9 deviation exactly like the vt runtime / the minja chat engine, and
delivers the user's explicit "GBNF as a vllm.cpp extension" mandate. The bitmask
INTERFACE (`[num_reqs, ceil(vocab/32)] int32`, bit==0 → -inf) is backend-invariant,
so the scheduler/runner/sampler plumbing is written once and never changes when
xgrammar later lands.

**Mirrored files:** `src/vllm/v1/structured_output/{backend_types,manager,request}.{h,cpp}`
(← `vllm/v1/structured_output/`), `backend_native.{h,cpp}` (the engine — original,
§9), `GrammarOutput` into `sched/output.h`, `Scheduler::get_grammar_bitmask` +
the accept_tokens advance into `sched/scheduler.cpp`, `apply_grammar_bitmask` into
the runner sample path, `StructuredOutputsParams` into `sampling_params.h`.

## Global Constraints

- 1:1 with pinned e24d1b24 for the INTEGRATION layer (StructuredOutputManager
  `__init__.py`, `backend_types.py`, `request.py`, scheduler `get_grammar_bitmask`
  :1477 + accept_tokens :1636, `GrammarOutput` output.py:262, `apply_grammar_bitmask`
  utils.py:85, EngineCore.step :492). Same class/method/field names. Ported files
  carry `// Ported from: vllm/v1/structured_output/<file> @ e24d1b24`. The native
  engine (backend_native) is ORIGINAL (§9).
- The bitmask semantic = the packed-int32 form of the M1.7 sampler's existing
  per-req allowed-token masking (`sampler.cpp` apply_allowed_token_ids /
  metadata.h allowed_token_ids_mask). Reuse that -inf masking.
- T0 StructuredOutputOptions = {JSON, JSON_OBJECT, REGEX, CHOICE, GRAMMAR}.
  DEFER (marked stubs, cite upstream): STRUCTURAL_TAG; the reasoning/thinking-mode
  `should_advance`/`should_fill_bitmask` gating (→ `return true` at T0); spec-decode
  multi-row bitmask; the async ThreadPoolExecutor compile (compile synchronously at
  T0); guidance/outlines backends.
- **THE BYTE-ALIGNMENT CORE (biggest risk):** the matcher must decide, per vocab
  token, whether its raw bytes can extend the current grammar state — handling
  byte-level BPE tokens (our tokenizer's GPT-2 bytes_to_unicode bijection),
  multi-byte UTF-8, special/added tokens, vocab holes. Build a token-byte trie ONCE
  at compile time (NOT O(vocab) per step). Wrong → invalid output; O(vocab)/step →
  kills decode throughput (bitmask fill runs every step per structured request).
- Warnings-as-errors; TDD; push authorized (CPU CI).

### Task 1: Structured-output type/contract skeleton
Read `vllm/v1/structured_output/backend_types.py` + `request.py` + the SamplingParams
structured fields. Port: `structured_output/backend_types.{h,cpp}` — `StructuredOutputOptions`

### Task 2: StructuredOutputManager + scheduler/EngineCore plumbing (backend-agnostic)
Read `vllm/v1/structured_output/__init__.py` (StructuredOutputManager) + scheduler
`get_grammar_bitmask` :1477 + accept_tokens :1636 + `GrammarOutput` output.py:262 +
core.py:492. Port: `structured_output/manager.{h,cpp}` — StructuredOutputManager
(grammar_init [SYNCHRONOUS compile at T0, no ThreadPoolExecutor], _create_grammar,
grammar_bitmask [the batched [max_reqs, ceil(vocab/32)] int32 fill; terminated/inactive
rows = -1 all-allowed], _fill_bitmasks; should_advance/should_fill_bitmask = return-true
stubs cite). `GrammarOutput` {structured_output_request_ids, grammar_bitmask} into
sched/output.{h,cpp}. `Scheduler::get_grammar_bitmask(scheduler_output) -> optional<GrammarOutput>`
(collect structured req-ids skipping prefill chunks → manager.grammar_bitmask) + the
`accept_tokens` FSM advance in `update_from_output` (after sampling, per structured req).
Call `get_grammar_bitmask` from `EngineCore::step` (mirror core.py:492 — pass to
sample_tokens). The manager holds ONE backend (constructed lazily on first grammar);
backend is Task 4. Until then grammar_bitmask over zero structured reqs → null.
Unit tests: a fake structured request flows through grammar_init → get_grammar_bitmask
produces a bitmask row; a non-structured batch → null; the accept_tokens advance is called.

### Task 3: apply_grammar_bitmask in the runner sample path
Read `vllm/v1/structured_output/utils.py::apply_grammar_bitmask` :85 + where the runner
calls it (gpu_model_runner.py:4462, before _sample). Port `apply_grammar_bitmask` into the
M1.8 runner's sample_tokens (src/vllm/v1/worker/gpu/runner.cpp): given the GrammarOutput
(the compacted bitmask + the structured req-ids) reorder rows to the batch/dense order
(spec offset = 0 at T0), unpack each int32 bitmask into the per-vocab mask, and mask the
logits to -inf (reuse the M1.7 sampler's allowed-token masking op, or a small
`vt::ApplyGrammarBitmask` sharing its -inf kernel). Runs BEFORE the sampler. The GrammarOutput
must reach sample_tokens (thread it through the EngineCore→executor→runner sample call —
mirror how upstream passes grammar_output to sample_tokens). Unit tests: a bitmask that
forbids all-but-token-K forces the sampled token to K; a null grammar_output is a no-op.

### Task 4: Native grammar engine backend (GBNF/EBNF + regex + choice → bitmask)
Implement `structured_output/backend_native.{h,cpp}` (ORIGINAL, §9) implementing the two
ABCs from Task 1: an EBNF/GBNF grammar parser → a compiled grammar (rule table) → a
per-request matcher with a stack-based FSM: `accept_tokens` (advance the FSM by the token's
bytes), `fill_bitmask` (for each vocab token, is it grammar-valid from the current state? →
set the bit), `rollback`/`is_terminated`/`reset`. **THE BYTE-ALIGNMENT CORE:** build a
token-byte TRIE from our tokenizer's byte-vocab ONCE at compile (tokenizer.h accessors +
the bytes_to_unicode decode); fill_bitmask walks the grammar × trie, NOT O(vocab) naive.
Handle byte-level tokens, multi-byte UTF-8, special/added tokens, vocab holes. Covers
GRAMMAR (EBNF/GBNF, llama.cpp-style) + REGEX (regex→grammar) + CHOICE (choice→grammar,
mirror utils.py choice_as_grammar). Wire the manager to construct this backend. Unit tests:
a GBNF grammar (e.g. a JSON-subset or a `("yes"|"no")` choice) + a token stream → the
bitmask forbids invalid continuations at each step + the accepted sequence validates + a
terminated grammar; the byte-trie correctness on multi-byte/special tokens; NOT O(vocab)
(a perf smoke: fill_bitmask time roughly independent of masked-out count).

### Task 5: JSON-schema → GBNF + `json_object` + OpenAI response_format wiring
